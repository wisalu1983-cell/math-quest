项目管理体系图

读项目时先进入 `CLAUDE.md` 与 `ProjectManager/Overview.md`；写项目时先改各自权威源，只有影响当前状态、主线或下一步时再回写 `Overview.md`。

01会话入口

CLAUDE.md / AGENTS.md

先读高频项目规则、执行适配和低频索引。它回答“本次该从哪里开始”。

ProjectManager/Overview.md

活跃控制面：只保留背景、当前版本轴、当前状态、下一步和入口链接。

先确认当前上下文，再决定读哪个权威源
02路由判断

Plan/README.md

计划索引、规则入口、模板入口。生命周期变化时才更新索引。

pm-write-routing.md

规定“先改权威源，再按影响回写 Overview.md”。

document-ownership.md

判断 Plan、Specs、Reports、QA 分别该承载什么。

按问题性质进入对应权威源
03权威源层

Plan/vX.Y/

版本执行事实。 README、00-04、phases、subplans 记录范围、阶段、证据和执行纪律。

Specs/

长期功能承诺。 `_index.md` 是新计划必查清单，`current.md` 只写验收后的行为。

ISSUE_LIST.md

具体 bug / 欠账。 生命周期是 open → closed，关闭项进版本归档。

Backlog.md

候选需求 / 方向。 候选 → 纳入版本 → 落地 / 延期 / 放弃。

实施或决策完成后，进入验证和沉淀
04验证沉淀

QA/runs/

正式 QA 结论入库；按风险选择 L1 / L2 / L3，release gate 使用 L3。

Reports/

复盘、机制说明、试点结果。默认不回写，除非产生新的活跃结论。

归档与同步门

版本启动、版本收口、多源生命周期变化时，按规则触发 `pm-sync-check`。

最后只做一次状态判断，不画回环线
05回写判断

回到 Overview.md 的唯一条件

只有当改动影响当前主线、当前状态、下一步或权威入口时，才回写 `ProjectManager/Overview.md`。否则信息留在对应权威源里。

版本开发流程图

正式产品版本从需求或缺陷被选中开始，先建立版本计划目录，再拆成阶段和具体任务推进。括号里保留 math-quest 的内部文件名，方便回到仓库定位。

01激活来源

需求池（Backlog）

记录还没排进版本的需求、想法和延期项。只有被选进某个版本后，才进入正式开发。

缺陷清单（Issue）

记录明确的 bug、技术欠账或实现问题。进入版本后保留原编号和复现证据。

真实用户反馈

先整理来源、用户痛点和期望改善，再决定是否纳入下一个版本。

决策进入某个正式产品版本
02版本启动

建立版本计划目录（Plan/vX.Y）

为这个版本放一套固定文档：入口、目标快照、来源证据、范围分类、阶段计划和执行规则。

核对长期规则（Specs/_index）

检查已有功能规则、体验约束和数据约定，把本版本必须遵守的内容写进计划。

建立阶段结构和实施入口
03阶段推进

阶段总图（Phase Plan）

把版本拆成几个阶段，说明先后顺序、依赖关系、开始条件和完成标准。

具体任务计划（subplan）

单个功能或修复开工前，先写清预期效果、调研证据、方案和验收方式。

独立开发环境（worktree）

涉及代码实现时，通常在独立工作区开发，避免影响主工作区里的其他改动。

每个阶段按风险选择验证深度
04验证与规格

基础验证

根据改动范围运行单元测试、构建检查、浏览器走查或人工体验检查。

分层 QA（QAleader）

普通功能做标准测试；高风险模块加深测试；版本发布前做完整回归和发布门禁。

规格影响判断

判断这次改动是否改变长期规则：需要更新、没有影响，或等验收后再确认。

版本完成后统一收束生命周期
05版本收口

整理需求和缺陷状态

已完成的需求和缺陷进入归档；没做完但仍值得做的，回到需求池或标为延期。

更新版本入口

把版本目标、完成事实、遗留风险和最终状态写回版本入口文档。

同步总览和索引

更新计划索引；如果当前版本或下一步变了，再更新项目总览并运行文档一致性检查。

进入下一轮候选评估
06当前落点

v0.4 已发布，开放 issue 为 0

后续新需求会先进入需求池或来自真实用户反馈；被选中后，再作为 `v0.5` 重新进入上面的版本启动流程。

介绍材料例外

本 HTML 是流程介绍材料，不属于产品版本内容。它不需要创建版本计划目录、不需要具体任务计划、不需要做规格影响判断，也不默认运行文档一致性检查。

速查矩阵

遇到不同问题时，先找对应的事实源，再判断是否需要更新项目总览或索引。

场景 先读 / 先改 主要职责 何时更新项目总览
查看当前项目状态 ProjectManager/Overview.md 项目背景、当前版本轴、当前状态、下一步、权威入口。 它本身就是控制面；不承载详细执行过程。
开新版本 / 版本收口 Plan/version-lifecycle.md + 版本包模板 创建或收束版本计划目录，维护入口、目标快照、阶段计划、具体任务计划和归档。 版本轴、当前阶段、当前状态或下一步变化时。
启动具体任务 Specs/_index.md + 相关版本计划 先核对已生效的长期规则，把本次必须遵守的约束写进任务计划。 实施结果改变活跃主线、状态或入口时。
修改长期功能承诺 Specs/_index.md 与功能当前规则页 功能当前规则页只写已经验收确认的长期行为；开发中的设想先留在任务计划里。 当前阶段入口或生效约束变化时。
处理 bug / 欠账 ISSUE_LIST.md 具体问题从“待处理”流转到“已关闭”；关闭项在版本收口进入归档。 影响当前主线、阻塞状态或下一步时。
处理候选需求 Backlog.md 需求从“候选”流转到“纳入版本 / 已落地 / 延期 / 放弃”。 候选被正式纳入、延期或影响版本决策时。
正式 QA / Release Gate QA/qa-leader-canonical.md + `QA/runs/` 按风险决定测试深度，记录自动化测试、人工体验检查、视觉检查和发布门禁结论。 QA 结论影响版本状态、阻塞项或发布判断时。
复盘 / 机制说明 Reports/ 保留历史机制、复盘、试点结果；默认不干扰活跃视图。 只有复盘产生新的活跃结论或入口变化时。
