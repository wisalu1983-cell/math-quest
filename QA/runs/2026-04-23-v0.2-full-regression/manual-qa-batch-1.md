# Manual QA Batch 1

**执行日期**：2026-04-23  
**层级**：`qa-leader` 第三层 · 拟真人工 QA  
**方法论**：`agent-as-user-qa` 四栏协议  
**目标用户画像**：上海五年级学生，数学能力中等，主要使用 375px 左右的小屏手机  
**范围**：`B-01 ~ B-04`、`C-01 ~ C-03`、`D-01 ~ D-03`

## 批次目标

- 确认新用户从建档到首局闯关、再到记录复盘的主路径自然可懂
- 确认 `v0.2-5-1`（F2 历史答题记录，本地版）在“完成 / 中止 / 逐题详情”三种关键场景下都可信
- 确认进阶训练、错题沉淀与段位赛 `BO3` 作为长期练习闭环，对目标用户来说路径明确、反馈稳定

## 前置条件

- 浏览器使用本地 dev server：`npm run dev -- --host 127.0.0.1 --port 5173`
- 不修改产品代码，只允许在浏览器端清空存档或注入受控夹具作为场景准备
- 每条用例单独留存截图证据；涉及状态切换的场景允许附多张前后对比图

## 用例清单

| ID | 用例名称 | 重点关注 |
|----|---------|---------|
| B-01 | 新号建档与首页 | 首次进入是否清晰，首页信息层级是否让孩子知道“先做什么” |
| B-02 | 首局闯关到结算 | 进入关卡、答题、结算节奏是否自然 |
| B-03 | 历史记录与逐题详情 | 完成后是否能顺畅回看刚做过的这一局 |
| B-04 | 主动退出生成未完成记录 | 中途退出后，记录是否既真实又不让人误解为已通关 |
| C-01 | 进阶入口锁态/解锁态 | 成长路径是否明确，锁定原因是否可理解 |
| C-02 | 进阶结算与错题本 | 练习后的正反馈、错题沉淀与个人中心联动是否成立 |
| C-03 | 刷新后持久化 | 刷新后是否保住昵称、进阶状态和记录，不让用户产生“白玩了”焦虑 |
| D-01 | Home / Hub 状态展示 | 段位赛门槛不足、可挑战、活跃赛事三态是否都能告诉用户下一步 |
| D-02 | BO3 晋级与淘汰 | 两局连胜/连败的竞技反馈与复盘反馈是否成立 |
| D-03 | 刷新恢复 | 局内与局间刷新是否都能继续，不丢赛事 |
