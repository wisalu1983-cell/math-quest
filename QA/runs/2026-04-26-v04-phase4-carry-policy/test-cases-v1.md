# v0.4 Phase 4 测试用例 v1

> 执行范围：`BL-005.2` 进位/退位格三档规则、`BL-003` compare tip 补证  
> 依据：`ProjectManager/Plan/v0.4/subplans/2026-04-26-phase4-进位退位格规则与compare-tip补证.md`  
> 目标用户：上海五年级学生，数学能力中等，移动端触摸与桌面键鼠都可能使用  
> 用例方法：参考 ISTQB test case specification 的“输入、执行条件、预期结果”结构，结合 ISO/IEC/IEEE 29119 的测试过程/设计技术口径；具体设计采用等价类、边界值、状态流、决策表、风险驱动和用户旅程覆盖。

---

## 方法映射

| 方法 | 本 Phase 使用方式 |
|---|---|
| 等价类 | 低档 / 中档 / 高档三档；答案格 / 过程格；空值 / 0 / 非 0 / 错值。 |
| 边界值 | `difficulty=5/6/7/8`，减法 `-` vs `-1`，乘法过程格 `0/8/9`。 |
| 决策表 | 答案正确性 × 过程格填写状态 × 难度档位 → `pass/fail/warning`。 |
| 状态流 | 填格 → 自动聚焦 → 提交拦截 → 本地复盘 → 统一结果 UI → 下一题/错题本。 |
| 用户旅程 | 小学生在低档学习步骤、中档尝试辅助格、高档直接挑战；移动端触摸和桌面键盘均覆盖。 |

## I — 竖式填空

设计意图：竖式板要让低档学生练到过程，中档学生可选使用过程格，高档学生不被过程格辅助。

| ID | 用例名称 | 操作 | 预期结果 | 预期体验 | 优先级 | 验证方式 |
|---|---|---|---|---|---|---|
| I-01 | 低档 999+888 自动跳到十位进位格 | 构造低档加法列：个位答案填 `7` | 下一焦点是十位进位格 | 学生按竖式步骤自然记录进位 | P0 | 自动化 |
| I-02 | 低档进位格填完跳同列答案格 | 在 I-01 后给十位进位格填 `1` | 下一焦点是十位答案格 | 不需要猜下一步在哪里 | P0 | 自动化 |
| I-03 | 低档 0 过程格显示但不必填 | 构造无进位低档加法，过程格期望 `0`，答案格全填，过程格留空提交 | 可提交且过程格按 0 判定 | 学生可自己判断“不需要进位” | P0 | 自动化 |
| I-04 | 低档 Enter/Tab 可跳过 0 过程格 | 当前焦点在期望 `0` 的过程格，按 `Enter` / `Tab` | 焦点移动到下一应填格 | 桌面端操作顺手，不被 0 格卡住 | P1 | 自动化 + 模拟人工 |
| I-05 | 低档非 0 过程格未填拦截提交 | 答案格全填，非 0 过程格留空，点击提交 | 不提交；返回 `missing-process` 并提示“请把进位/退位格填完整” | 提示明确，不误以为按钮坏了 | P0 | 自动化 + 模拟人工 |
| I-06 | 低档过程格错判未通过 | 答案格正确，非 0 过程格填错，提交 | `failProcess`，原因 `vertical-process` | 孩子知道是过程错，不是答案错 | P0 | 自动化 + 模拟人工 |
| I-07 | 减法 `-` 不算完整 | 当前焦点在退位格，输入 `-` | 不跳格，值保持待完成状态 | 不会半截输入就跳走 | P0 | 自动化 |
| I-08 | 减法 `-1` 算完整 | 当前焦点在退位格，输入 `-` 再输入 `1` | 退位格完成，触发下一焦点 | 退位输入符合直觉 | P0 | 自动化 |
| I-09 | 乘法过程格只接受 0-8 | 构造乘法过程格，分别输入 `8` 和 `9` | `8` 完整；`9` 不作为有效过程值 | 边界值稳定，不接受不可能进位 | P0 | 自动化 |
| I-10 | 中档答案格默认跳答案格 | 构造中档加法，答案格填完一位 | 下一焦点是下一答案格，不进入过程格 | 中档不被辅助格打断 | P0 | 自动化 |
| I-11 | 中档主动填过程格后跳同列答案格 | 手动点击中档过程格并填完整 | 下一焦点为同列答案格 | 主动记录过程的学生得到顺手引导 | P0 | 自动化 + 模拟人工 |
| I-12 | 高档不显示过程格 | 构造高档竖式题 | `visibleCells` 不含过程格 | 高档界面更简洁，更像挑战 | P0 | 自动化 + 模拟人工 |

## D — 答题通用反馈

设计意图：所有结果都走统一判定 UI；中档过程错误只做当前提示，低档过程错误走失败和扣心。

| ID | 用例名称 | 操作 | 预期结果 | 预期体验 | 优先级 | 验证方式 |
|---|---|---|---|---|---|---|
| D-01 | 低档 `failProcess` 扣心 | 低档答案正确、过程格错误，完成提交链路 | `lastAnswerCorrect=false`，心数 -1 | 低档过程训练有明确边界 | P0 | 自动化 |
| D-02 | 低档 `failProcess` 统一失败 UI | 点竖式板本地复盘后的继续 | 进入统一失败结果 UI，显示未通过原因 | 流程和普通错题一致，不突兀 | P0 | 模拟人工 |
| D-03 | 中档 `passWithProcessWarning` 不扣心 | 中档答案正确、已填过程格错误 | `lastAnswerCorrect=true`，心数不变 | 学生被温和提醒，不被惩罚 | P0 | 自动化 |
| D-04 | 中档过程提示只在统一 UI | 中档答案正确、过程格错误，观察竖式板和结果 UI | 竖式板不标红/不黄块；结果 UI 有当前题过程提示 | 提醒位置统一，不干扰答题板 | P0 | 模拟人工 |
| D-05 | 中档过程格留空无提示 | 中档答案正确，过程格全留空 | `pass`，无过程提示 | 不使用辅助格不会被打扰 | P0 | 自动化 + 模拟人工 |

## E — 错题本 / 历史 / 同步

设计意图：低档过程失败进入错题本并显示原因；中档过程提示不持久化。

| ID | 用例名称 | 操作 | 预期结果 | 预期体验 | 优先级 | 验证方式 |
|---|---|---|---|---|---|---|
| E-01 | 低档过程失败进入 pendingWrongQuestions | 调用 store 提交 `failProcess` | `pendingWrongQuestions[0].failureReason='vertical-process'` | 错题归因可追溯 | P0 | 自动化 |
| E-02 | endSession 后错题本保留过程失败原因 | 结束含 `failProcess` 的 session | `gameProgress.wrongQuestions` 保留 `failureReason` | 错题本能解释为什么没通过 | P0 | 自动化 |
| E-03 | 错题本显示过程失败原因 | 打开错题本查看该题 | 显示“未通过原因：进位/退位格填写错误” | 家长/孩子看得懂错因 | P1 | 模拟人工 |
| E-04 | 中档过程 warning 不进错题本 | 中档 `passWithProcessWarning` 后结束 session | 不新增 wrong question；history 不写 failureReason | 只保留当前提醒，不污染复习 | P0 | 自动化 |
| E-05 | sync merge 不丢 failureReason | 合并 local/remote wrongQuestions，其中一条带原因字段 | 合并后字段保留 | 跨设备同步后错因不丢 | P0 | 自动化 |

## F — 题目生成 / 方法提示

设计意图：compare 概念题在目标难度能稳定展示方法提示，Boss 仍不显示。

| ID | 用例名称 | 操作 | 预期结果 | 预期体验 | 优先级 | 验证方式 |
|---|---|---|---|---|---|---|
| F-01 | d=8 multi-select compare 有 tip | 用可控题对象调用 `getMethodTip()` | 返回“遇到‘一定’，先找反例” | 高档概念判断前有方法抓手 | P0 | 自动化 |
| F-02 | d=8 判断正误 compare 有 tip | 用可控题对象调用 `getMethodTip()` | 返回 compare tip | 二选一概念判断也能看到提示 | P0 | 自动化 |
| F-03 | d=9 Boss compare 无 tip | 用可控题对象调用 `getMethodTip()` | 返回 `null` | Boss 保持挑战感 | P0 | 自动化 |
| F-04 | 浏览器补证 compare tip 可见 | dev 模式锁定或注入 d<=8 compare 题，进入 Practice | 题干下方可见 compare tip | 孩子真实页面上能看到提示 | P0 | 模拟人工 |
