## 1. 版本定性

| 版本 | 主题 | 性质 |
|---|---|---|
| v0.1 | 三层游戏化闭环 | 大功能落地 |
| v0.2 | 用户反馈驱动题目质量升级 | 体验打磨 |
| v0.3 | Supabase 账号同步 | 基础设施 |
| **v0.4** | **题目体验系统性修复** | **用户反馈落地 + 技术债清理** |

**定性结论**：v0.4 以 4.25 真实用户反馈为主驱动，系统性解决竖式笔算、运算律 UX、选项题设计和重复题目四大体验问题，同时清理 v0.2 遗留技术债（BL-003/BL-004/ISSUE-059）。

| 子项 | 改动性质 | Phase |

## 4. Phase 结构草案

### Phase 1 — 渲染与判定修复（低风险，快速落地）

**目标**：修复视觉和判定层问题，不涉及生成器或复杂设计决策。

| 任务 | 来源 |
|---|---|
| 竖式颜色：提升数字/符号对比度 | BL-005.1 |
| 小数答案：兼容末尾不写零/小数点的写法 | BL-005.5 |
| 乘法竖式：统一交互形式（调研后选定一种） | BL-005.3 |
| dec-div 高档：清除隐藏 trainingFields | ISSUE-059 |

**收口条件**：所有修复项有 E2E 或手工验证覆盖；`npm test` / `npm run build` 通过。

### Phase 2 — 题目质量（生成器/内容层）

**目标**：解决题目本身的设计问题——难度匹配、选项丰富度、重复去重。

| 任务 | 来源 |
|---|---|
| 进阶3星难度排查：360÷3 类题剔除（查生成器难度分档逻辑） | BL-005.4 |
| 选项题：扩充干扰项至3~4个，加概念混淆型 | BL-007 |
| 重复题目：排查各模式各题型去重逻辑是否生效（先诊断再修） | BL-008 |

**注意**：BL-005.4 涉及生成器，开工前必查维度 A 规格（`generator-redesign-v2.md` + `generator-difficulty-tiering-spec.md`）和维度 B（`TOPIC_STAR_CAP`）。

**收口条件**：难度修复有题目抽样验证（同 v0.2 的生成器 QA 方式）；BL-008 有明确 bug/设计结论并已处理。

### Phase 3 — 交互设计与教学引导（设计变更）

**目标**：处理需要设计决策的 UX 问题，以及教学提示补证。

| 任务 | 来源 |
|---|---|
| 进位格判定逻辑：明确三档规则，实现区分处理与结果提示 | BL-005.2 |
| 运算律 UX：去除槽位"1/2"序号 + 补充操作说明文字 | BL-006 |
| compare tip 补证：专项抽样验证可达性 | BL-003 |

**注意**：BL-005.2 的进位格逻辑需要产品决策（三档规则具体设计），Phase 3 开工前需先确认规则。

**收口条件**：BL-006 有用户视角走查；BL-005.2 有三档场景手工验证。

### Phase 4 — 工程质量（技术债）

**目标**：清理实现层欠账，降低后续维护成本。

| 任务 | 来源 |
|---|---|
| Practice 答题页：重构状态重置为统一单一机制 | BL-004 |

**收口条件**：重构前后行为等价，有回归测试覆盖。
