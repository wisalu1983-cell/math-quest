# LLM 出题功能预研报告

> 创建：2026-04-25
> 性质：纯调研 / 预研，不挂靠任何版本
> 状态：草稿

---

## 1. 功能概述

向通过 VIP 认证的账号开放 LLM 辅助出题能力。联网时自动激活，离线或 LLM 服务不可用时无感退回现有算法生成器。

**核心约束**：

- VIP 专属，非 VIP 用户无感知
- 联网自动激活，无需用户手动切换
- 离线/失败时静默回退到基础生成器，不阻塞答题流程

## 2. 产品经理视角

### 2.1 用户价值

| 维度 | 基础生成器（现状） | LLM 增强（预期） |
|------|-------------------|-----------------|
| 题目表述多样性 | 模板池有限，重复度高 | 同一考点可生成数十种不同措辞 |
| 错误诊断题质量 | 硬编码错误步骤，套路固定 | 动态生成仿真错误链，每次不同 |
| 情境题覆盖面 | 手工编写场景池，数量瓶颈 | 按年龄段动态生成贴近生活的场景 |
| 干扰项质量 | 随机组合，部分选项明显不合理 | 模拟学生真实错误思路，提升辨别训练价值 |
| 概念辨析深度 | 固定问法，学生靠记忆应对 | 多角度包装同一陷阱，迫使真正理解 |

### 2.2 各题型收益预估

| 题型 | 收益等级 | 受益子题型 | 说明 |
|------|---------|-----------|------|
| A07 简便计算 | ★★★★★ | error-diagnose, 伪简便辨析 | 错误链生成 + "看似能简便"的反例构造，当前模板池最易耗尽 |
| A04 运算律 | ★★★★★ | error-diagnose, counter-example, easy-confuse, concept-reverse | 概念推理类子题型占比高，phrasing 多样性是核心瓶颈 |
| A08 方程移项 | ★★★★ | error-diagnose, equation-concept | 四类 trap（T1-T4）的错误步骤链需要更多变体 |
| A02 数感估算 | ★★★★ | estimate（生活情境） | 进一法/去尾法场景高度依赖语言多样性 |
| A05 小数性质 | ★★★ | trap, compare, cyclic-div | 反直觉概念题受益，但纯计算部分不需要 LLM |
| A06 括号变换 | ★★ | error-diagnose | 仅诊断子题型受益，核心表达式变换不需要 |
| A01 基础计算 | ★ | — | 纯计算，LLM 无优势且有算错风险 |
| A03 竖式笔算 | ★ | — | 逐列步骤生成是精确算法工作，LLM 不适用 |

### 2.3 功能设计要点

**VIP 认证**：

- 需要新增用户层级概念（当前 User 类型无任何 tier/subscription 字段）
- VIP 认证方式待定：邀请码 / 管理后台标记 / 付费订阅
- VIP 状态需持久化到 Supabase `profiles` 表，本地同步

**激活与回退体验**：

- 用户不应感知到"两套出题引擎"的存在
- 激活/回退不应产生视觉跳变或加载中断
- 回退时题目质量下降是可接受的（因为基础生成器本身就是完整功能）
- 需考虑：同一次练习中途断网，已用 LLM 生成的题和回退后的题是否混合出现

**度量与反馈**：

- 需要埋点区分 LLM 题 vs 基础题的用户表现数据（正确率、用时、跳过率）
- 用于验证 LLM 出题是否真的提升了学习效果，而非只是"看起来更好"

## 3. 技术负责人视角

### 3.1 现有架构现状

| 维度 | 现状 | 对 LLM 集成的影响 |
|------|------|------------------|
| 生成器接口 | 纯同步函数 `(GeneratorParams) -> Question` | LLM 调用是异步的，需要适配层 |
| 调度器 | `generateQuestion(topicId, difficulty, subtypeFilter)` 同步返回 | 需要异步版本或预生成策略 |
| 网络层 | 仅 Supabase 客户端，无通用 HTTP 服务层 | 需新建 LLM API 服务模块 |
| 离线检测 | sync engine 已有 `navigator.onLine` + 事件监听 | 可复用，但需扩展到 LLM 服务可用性检测 |
| 用户模型 | User 无 VIP 字段，无订阅/层级概念 | 需扩展 User 类型 + Supabase schema |
| 存储 | 本地优先 localStorage + Supabase 同步 | LLM 生成的题目是否需要缓存？ |

### 3.2 架构方案概要

```
┌─────────────────────────────────────────────────┐
│                  答题流程                         │
│  generateQuestion(topicId, d, filter)            │
└──────────────────┬──────────────────────────────┘
                   │
          ┌────────▼────────┐
          │  LLM 路由决策层   │
          │  isVIP?          │
          │  isOnline?       │
          │  isLLMHealthy?   │
          └───┬─────────┬───┘
              │         │
     ┌────────▼──┐  ┌───▼────────┐
     │ LLM 生成   │  │ 基础生成器   │
     │ (异步预取)  │  │ (同步,现有) │
     └────────┬──┘  └───┬────────┘
              │         │
          ┌───▼─────────▼───┐
          │  统一 Question    │
          │  输出 + 校验层    │
          └─────────────────┘
```

**关键设计决策**：

#### 方案 A：实时调用

每次需要题目时实时调用 LLM API，带超时回退。

- 优势：实现简单，始终获得最新结果
- 劣势：**延迟不可控**（LLM API 响应 1-5s），用户在答题间隙会感受到等待；依赖网络质量

#### 方案 B：预生成缓冲池

后台预生成一批题目缓存到本地，答题时从缓冲池取题。

- 优势：零感知延迟，离线后缓冲池中剩余题目仍可使用
- 劣势：实现复杂（缓冲池管理、过期策略、难度/题型分布预测）；首次使用前需预热

#### 方案 C：混合模式（推荐评估方向）

- 首次进入练习时预取 5-10 题到缓冲池
- 答题过程中后台逐步补充
- 缓冲池耗尽时回退到基础生成器
- 离线时消耗缓冲池存量，用完后回退

### 3.3 LLM 调用层设计

**Prompt 工程**：

- 每个题型需要专用 system prompt，包含：该题型的数学约束、难度定义、子题型分布、输出 JSON schema
- 必须输出结构化 JSON（与现有 `Question` 接口兼容）
- 需要 few-shot 示例确保格式稳定性

**校验层**（不可省略）：

LLM 生成的每道题必须经过独立校验，至少包括：

| 校验项 | 方法 | 失败处理 |
|--------|------|---------|
| JSON schema 合规 | Zod / JSON Schema 校验 | 丢弃，回退基础生成器 |
| 数学正确性 | 独立计算器验算 answer 是否匹配 prompt | 丢弃 |
| 难度档位匹配 | 规则引擎检查操作数范围、步骤数 | 降级或丢弃 |
| 干扰项合理性 | 选项不重复、不与正确答案相同 | 自动修复或丢弃 |
| 表达式可解析 | LaTeX / 表达式解析器检查 | 丢弃 |
| 内容安全 | 基础敏感词过滤 | 丢弃 |

**校验失败率预估**：根据行业经验，初期 prompt 未调优时校验失败率可能在 15-30%，调优后可降至 5-10%。这意味着预生成方案需要额外冗余请求。

### 3.4 实现工作量估算

| 模块 | 工作内容 | 估算人天 | 难度 |
|------|---------|---------|------|
| User 模型扩展 | User 类型 + Supabase schema + 迁移链 + VIP 状态同步 | 2-3d | 低 |
| LLM 服务模块 | API 客户端、重试、超时、健康检查 | 3-4d | 中 |
| Prompt 工程 | 8 题型 × prompt 设计 + few-shot + 调优迭代 | 5-8d | 高 |
| 校验层 | Schema 校验 + 数学验算 + 难度检查 | 4-5d | 高 |
| 缓冲池管理 | 预生成、存储、过期、分布平衡 | 3-5d | 中 |
| 路由决策层 | VIP 检测 + 在线检测 + 健康检查 + 回退逻辑 | 2-3d | 中 |
| 生成器适配 | 现有 8 个生成器的异步包装 + 统一出口 | 2-3d | 低 |
| 埋点与度量 | 区分题源、采集表现数据 | 1-2d | 低 |
| 测试 | 单元测试 + 集成测试 + 离线场景 + 回退场景 | 3-5d | 中 |
| **合计** | | **25-38d** | |

> 注：prompt 工程的工作量波动最大，取决于 LLM 输出稳定性和校验通过率。上表以优先覆盖第一梯队题型（A04、A07）为基线；全题型覆盖需额外 50% 工作量。

### 3.5 运行成本估算

以 Claude Sonnet 4 为参考（假设单次调用 ~1K input tokens + ~500 output tokens）：

| 场景 | 单价假设 | 日均调用量 | 月成本 |
|------|---------|-----------|--------|
| 10 个 VIP 用户，日均 30 题 | $3/M input, $15/M output | 300 次 | ~$3-5 |
| 100 个 VIP 用户，日均 30 题 | 同上 | 3,000 次 | ~$30-50 |
| 1000 个 VIP 用户，日均 30 题 | 同上 | 30,000 次 | ~$300-500 |
| 带校验冗余（×1.3） | 同上 | ×1.3 | ×1.3 |

> 若使用缓冲池 + 题目缓存复用，实际调用量可压缩 30-50%。
> 使用更小的模型（如 Haiku）可再降 5-10 倍成本，但题目质量需实测验证。

### 3.6 LLM 供应商选择考量

| 维度 | 自建代理 | 直连 Claude API | 直连 OpenAI API | 国内模型（DeepSeek / 通义等） |
|------|---------|----------------|----------------|---------------------------|
| 延迟（中国大陆） | 可优化 | 需翻墙或代理 | 需翻墙或代理 | 低 |
| 合规 | 自控 | 数据出境风险 | 数据出境风险 | 合规友好 |
| 数学能力 | 取决于底层模型 | 强 | 强 | 中等（需实测） |
| 成本 | 部署成本 + API 成本 | 中 | 中 | 低 |
| 结构化输出稳定性 | 取决于底层模型 | 好 | 好 | 需实测 |

## 4. 风险评估

### 4.1 高风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| **LLM 生成数学错误的题目** | 学生学到错误知识，信任崩塌 | 独立校验层必须覆盖所有数学断言；校验失败时静默丢弃 |
| **延迟导致答题卡顿** | 用户体验劣化，VIP 反而比免费差 | 缓冲池预取；超时阈值（如 3s）后立即回退 |
| **Prompt 注入 / 不当内容** | 生成含不当信息的题目 | 输出过滤 + 结构化 JSON 约束 + 内容审核 |

### 4.2 中风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 校验失败率过高 | 实际可用题目产出率低，回退频繁，VIP 价值感弱 | 迭代 prompt；接受初期仅覆盖高收益题型 |
| API 供应商不稳定 / 限流 | 服务间歇性不可用 | 多供应商 fallback；缓冲池提供缓冲窗口 |
| 成本随用户增长线性上升 | 财务不可持续 | 题目缓存复用；低频题型不走 LLM；设置每日调用上限 |
| 现有生成器同步接口被破坏 | 全局回归风险 | 适配层包装，不改动现有生成器内部逻辑 |

### 4.3 低风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| VIP 用户数不足以覆盖成本 | ROI 为负 | 小规模验证后再决定是否扩大 |
| 离线回退后题目质量落差明显 | 用户困惑 | 基础生成器持续迭代，缩小差距 |

## 5. 待决策的疑问

### 5.1 产品决策

| # | 问题 | 影响范围 | 建议 |
|---|------|---------|------|
| P1 | VIP 认证方式：邀请码 / 管理后台手动标记 / 付费订阅？ | User 模型设计、Supabase schema | 初期用邀请码或后台标记，降低复杂度 |
| P2 | LLM 出题覆盖全部 8 题型还是仅覆盖高收益题型？ | prompt 工程工作量、成本 | 先覆盖 A04 + A07，验证效果后再扩展 |
| P3 | 用户是否需要知道当前题目来自 LLM？ | UI 设计、信任管理 | 建议不区分展示，避免形成"LLM 题更难/更好"的心理预期 |
| P4 | 同一次练习中途回退，是否需要通知用户？ | 交互设计 | 建议不通知，保持无感 |
| P5 | VIP 到期后，本地缓冲池中已有的 LLM 题是否仍可使用？ | 缓冲池策略 | 建议可使用（已生成的题目不回收） |
| P6 | LLM 题和基础题的练习历史是否需要区分标记？ | 数据分析、存档 schema | 建议在 historyRecord 中增加 source 字段，用于内部分析 |

### 5.2 技术决策

| # | 问题 | 影响范围 | 建议 |
|---|------|---------|------|
| T1 | LLM API 调用从前端直连还是经 Supabase Edge Function 代理？ | 安全性、API key 暴露 | **必须后端代理**——前端不可暴露 API key |
| T2 | 缓冲池存储在 localStorage 还是 IndexedDB？ | 存储容量、性能 | IndexedDB，localStorage 有 5-10MB 限制 |
| T3 | 校验层运行在前端还是后端？ | 延迟、安全、复杂度 | 后端校验（Edge Function 内），减少无效传输 |
| T4 | 存档版本号是否需要 bump？LLM 题源字段是否需要迁移链？ | 存档兼容性 | 如果 Question 增加 `source` 字段，需要迁移 |
| T5 | 选择哪个 LLM 供应商？是否需要支持多供应商切换？ | 架构复杂度、合规 | 先选一个验证，接口层预留切换能力 |
| T6 | 超时阈值设多少？回退后是否记录失败原因？ | 用户体验、运维 | 3s 超时；记录失败原因到日志表用于优化 |
| T7 | 现有 `generateQuestion` 入口是否改为异步？ | 全局调用点改动范围 | 可用异步包装层，现有同步调用点无需修改 |

## 6. 可行性结论

| 维度 | 评估 |
|------|------|
| 技术可行性 | **可行**。核心难点在 prompt 工程和校验层，无不可逾越的技术障碍。现有架构的离线检测、同步引擎可复用。 |
| 产品可行性 | **可行但需验证**。LLM 对高阶概念题的提升有理论基础，但"更好的题目"是否转化为"更好的学习效果"需要 A/B 数据验证。 |
| 成本可行性 | **小规模可行**。10-100 VIP 用户的月成本在 $5-50，可接受。千级用户需要缓存策略和成本优化。 |
| 时间可行性 | **中等**。最小可用版本（仅 A04 + A07 + 基础回退）约 15-20 人天；全题型覆盖 35-40 人天。 |

## 7. 建议的验证路径

如果决定推进，建议分三步：

### Step 1：Prompt POC（3-5 天）

- 选择 A07 `error-diagnose` 一个子题型
- 手工编写 prompt + few-shot，用 Playground 验证输出质量
- 统计：格式合规率、数学正确率、难度匹配率
- 产出：go/no-go 判断 + prompt 模板

### Step 2：端到端原型（7-10 天）

- 搭建 Supabase Edge Function 代理
- 实现缓冲池 + 回退逻辑
- 仅 A07 一个题型接入
- 不做 VIP 门控，开发者本地测试
- 产出：可运行的原型 + 延迟/成本/失败率实测数据

### Step 3：VIP 体系 + 扩展（10-15 天）

- User 模型扩展 + VIP 认证流程
- 接入第二个题型（A04）
- 埋点 + A/B 对比数据采集
- 产出：可灰度发布的完整功能

## 8. 附录：与现有版本规划的关系
