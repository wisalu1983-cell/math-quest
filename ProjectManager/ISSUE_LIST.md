## 开放问题清单

### ISSUE-067 · 多行乘法竖式判错面板缺少过程 / 训练格错因

- **优先级**：P1
- **类别**：bug / UX 反馈一致性 / A03 竖式计算
- **所属版本**：v0.5（Phase 3 输入与反馈基础设施）
- **来源**：2026-04-28 用户在本地测试环境验证多位乘法竖式后反馈
- **问题现象**：多位乘法竖式题中，用户把部分积等竖式过程格，或小数乘法复用竖式中的整数化 / 小数位数等训练格填错，但最终积填写正确时，系统会判错；统一失败面板只显示“正确答案”和题目解析，没有说明“竖式过程 / 训练格填写错误”。只看该面板，用户会认为“答案明明写对了，为什么算错”。
- **复现样例**：`110 × 43 = 4730`，最终积填 `4730`，但中间部分积或相关训练格填错；提交后进入失败反馈面板，面板只显示正确答案 `4730` 与解析 `110 × 43 = 4730`。
- **当前证据**：
  - `src/components/MultiplicationVerticalBoard.tsx` 当前提交只返回 `boolean`，缺少结构化错因。
  - `src/pages/Practice.tsx` 对多行乘法竖式收到 `false` 后提交 `"竖式计算有误"`，但没有传入可供统一失败面板展示的 `failureReason`。
  - 统一失败面板目前只对 legacy 单行竖式的 `lastFailureReason === 'vertical-process'` 展示“进位/退位格填写错误”，没有覆盖多行乘法部分积 / 小数乘法训练格错误。
- **用户影响**：判定本身正确，但反馈解释不足，容易被玩家理解为判题 bug；后续 v0.5 若新增竖式除法 UI 且采用“过程即答案”的严格判定，这类反馈缺口会放大。
- **初步修复方向**：让多行乘法竖式提交失败时携带结构化错因，统一失败面板展示明确文案，例如“未通过原因：竖式过程 / 训练格填写错误，虽然最终答案正确，但本题要求过程也正确”。如能低风险实现，可同时展示错误格数量或错误类型；具体字段设计在 v0.5 子计划中确认。
- **非目标**：不改变多行乘法竖式的判定规则；最终积正确但过程 / 训练格错误仍不通过。
- **验证要求**：覆盖整数多位乘法“部分积错、最终积对”和小数乘法复用竖式“训练格错、最终答数对”两类用例；移动端反馈面板文案不溢出。
