## Phase 3 目标与非目标

### 目标

把 v0.3 账号同步从"底层可用"推进到"用户安全可用、可理解、可验收"：

1. 登录入口 —— Onboarding / Profile 可进入邮箱 Magic Link 登录
2. 账号区域 —— Profile 展示邮箱、登录状态、同步状态、登出
3. 同步状态 —— Home 轻量展示、Profile 完整状态文字
4. 首次登录合并 —— 保护本地进度，确定数据归属
5. 账号切换 —— 不同账号数据严格独立，切换前确认当前账号已同步
6. 段位赛联网限制 —— 按 phase-3.md 产品规则 + D1 决策落地
7. Phase 2 4 条 RISK 消化 —— 每条给出结论（已修复 / 按产品决策保留 / 降级关闭）
8. 真实 Supabase 验收 —— 8 个剧本覆盖 Magic Link、跨设备合并、离线恢复、段位赛联网、合并失败、登出保护

### 非目标（明确不在 Phase 3 内）

- v0.2 Backlog（compare 方法提示补证、Practice 状态重置清理）
- 同步管理中心 / 同步详情页 / 手动同步控制台
- 密码登录、排行榜、好友、后台看板
- Practice 答题页同步状态展示
- 云端历史容量上限（v0.3 数据量在健康范围，观察到真实风险再做）

## 已定决策清单

以下 12 项决策（产品 5 项 + 技术 7 项）已在 Phase 3 开工前对齐。开发文档按这份清单展开，不再反复讨论。

### 产品 / 体验决策（来自 D1~D5 讨论）

| ID | 决策 | 理由 |
|---|---|---|
| P1 | 段位赛离开 Practice 页自动 `suspendActiveMatch()`；远端 `rank_match_sessions.updated_at` 超过 **10 分钟**视为可接管；未 suspend 且 10 分钟无响应，另一台设备可直接继续打，不需要用户手动"放弃" | 方案 C：拉平 active/suspended 的用户语义，覆盖主流云存档产品做法；10 分钟阈值覆盖绝大多数"临时离开"场景，同时避免用户换设备被长时间阻塞 |
| P2 | 已登录但持续离线时保持登录态，直接进入主场景，用 `SyncStatusIndicator` 显示"离线"；联网后才触发合并判定；一次 session 内不反复弹 banner | 对齐 Duolingo / Firebase Auth / local-first 2026 主流范式；避免 Logseq 反面教材的"反复告警" |
| P3 | 合并对话框失败态允许手动重试 + 换方向（合并 ↔ 使用云端）+ 取消登录三个按钮；**不提供"稍后再说"** | 阻塞 modal 必须有出口但不能有"中间态"；取消登录等价于 Supabase `signOut()` 回到访客主场景，数据归属清晰（对齐 VS Code / Pulsar 的 Cancel 语义 + Apple ID 切换失败时的 Sign Out 路径） |
| P4 | signOut 前有未同步数据时弹知情对话框，两按钮："取消"（主）+ "仍然登出"（次）；不提供"立即尝试同步"按钮（`SyncEngine` 已在后台自动重试）。用户确认"仍然登出"后，清空当前账号的待同步队列（`dirtyKeys`），本地副本保留在设备但不再带入下一账号 | 对齐 iCloud / Chrome 登出警告；尊重用户知情后的自主决定，同时避免旧账号未同步队列污染新账号 |
| P5 | 段位赛生命周期统一靠 `status` 字段表达（`active / suspended / cancelled / completed`），**不做跨端物理删除**；`deleteRankMatchSession` 仅供 dev-tool / 测试调用，产品路径无调用点；RISK-2 核实后降级关闭 | 对齐 Xbox GDK / PlayFab 游戏存档同步范式；段位赛历史是用户长期资产，云端应保留 |

### 技术决策（tech lead 拍板）

| ID | 决策 | 落点 |
|---|---|---|
| T1 | 沿用单 key 本地存储；`mq_auth_user_id` 作为"本地数据归属哪个 Supabase uid"的锁；不引入 per-account namespace | 文档 01 |
| T2 | `engine.ts` 改为 `arm(userId) → start() → shutdown()` 三态；合并判定完成前只 arm 不 start，禁止 `fullSync` 抢跑 | 文档 01 |
| T3 | 新增 `src/sync/local-progress.ts` 纯函数 `hasMeaningfulLocalProgress(gp: GameProgress): boolean`；是三态路由的唯一判据，不是防御性冗余 | 文档 01 |
| T4 | 指数退避常量 `RETRY_DELAYS = [1s, 2s, 4s, 8s, 16s, 30s]` / `MAX_RETRY = 6` 在 Task 3.0 提前导出；Task 3.5 实现退避行为；超上限 `status` 停留 `error`，不清 `dirtyKeys` | 文档 01 / 文档 04 |
| T5 | 自愈触发器：`online` 事件 / 下一次 `markDirty` / Realtime 推送 / 30s 轮询，任一成功即复位 `retryCount = 0` | 文档 04 |
| T6 | 段位赛跨设备接管：B 设备直接在本地写入合并后的 session 继续打，不修改 status；通过 `mergeRankMatchSessions`（取 games 更长者 + 状态优先级）保证覆盖顺序；A 设备若还在线会通过 Realtime 拉到 B 的新写入 | 文档 03 |
| T7 | 首次登录合并对话框内的单次远端 fetch/upsert 超过 **15s** 视为失败，走错误态；其他同步操作不加超时 | 文档 01 |

## RISK-1~4 在 Phase 3 的落点

来自 Phase 2 Code Review（见 [`../../Plan/v0.3/phases/phase-3.md#预研与遗留风险待办`](../../Plan/v0.3/phases/phase-3.md)）。

| ID | 来源现象 | Phase 3 处理 | 落点 |
|---|---|---|---|
| RISK-1 | `shutdown` 清空 `dirtyKeys`，登出 → 登录往返后脏标记丢失 | **修复**：`shutdown()` 常规路径不再清 `dirtyKeys`；signOut 前校验 dirtyKeys；有未同步数据时先走 P4 的知情对话框；用户确认"仍然登出"后才清空当前账号待同步队列，防止旧账号脏标记跨账号残留 | 文档 01 Task 3.0 + 文档 02 Task 3.3 |
| RISK-2 | `deleteRankMatchSession` 不推远端 | **降级关闭**：核实无产品路径调用者（仅 dev-tool / 单测），按 P5 决策补文档 + doc comment | 文档 03 Task 3.4（doc comment） |
| RISK-3 | `push` 失败无指数退避定时器 | **修复**：按 T4 / T5 实现 `RETRY_DELAYS` 定时重试 + 自愈 | 文档 04 Task 3.5 |
| RISK-4 | `mergeRankMatchSessions` 测试未覆盖对称用例 | **修复**：在 `src/sync/merge.test.ts` 补"同优先级 + 本地 games 更长"的对称用例 | 文档 04 Task 3.5 |

Phase 3 收尾报告需要逐条写明每条 RISK 的最终处理结果。

## 旧 `implementation-plan.md` Phase 3 草案的废弃说明

[`../../Plan/v0.3/implementation-plan.md`](../../Plan/v0.3/implementation-plan.md) 文件中 **Task 3.1 ~ Task 3.5（L1430-L1774）** 为 Phase 3 的原始草案。该草案在预研收口前编写，存在以下问题，**已被本文档集整体替代**：

1. 没有处理 SyncEngine 启动门控（直接在合并前允许 `fullSync` 抢跑）
2. 没有 `hasMeaningfulLocalProgress` 判定；合并对话框触发条件不准确
3. 没有账号归属锁 `mq_auth_user_id` 的写入时机
4. 没有处理 signOut 前未同步数据的保护
5. 没有离线启动 / 持续离线的体验设计
6. 没有指数退避、自愈、合并对话框错误态
7. 段位赛联网检查仅一行 `if (!navigator.onLine) throw`，缺远端活跃判定 + 10 分钟接管

处理方式：**`implementation-plan.md` 保留不删**（作为设计演进记录），在文件顶部添加"Phase 3 部分已废弃，以 `Specs/v03-supabase-account-sync/2026-04-24-phase3-*` 为准"的声明。Phase 1 / Phase 2 的 Task 1.x / 2.x 部分**继续有效**，作为历史实施记录存档。
