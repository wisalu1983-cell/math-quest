# v0.3 设计规格：Supabase 在线账号与数据同步系统

> 创建：2026-04-23
> 状态：**草稿**
> 版本：v0.3（最小可用：账号 + 同步 + 本地迁移）

---

## 1. 目标与范围

### 目标

为 math-quest 引入在线账号系统，实现跨设备（PC / 手机 / 平板）数据互通。用户在任何设备上登录同一账号，练习进度、段位、历史记录自动同步。

### 范围内

- 邮箱 Magic Link 无密码登录（Supabase Auth）
- 本地优先 + 后台同步架构（离线可用，联网后自动同步）
- 多设备离线冲突的确定性合并
- 首次登录时本地进度引导合并
- 访客模式保留（不登录 = 现状不变）
- 数据隔离（RLS，用户只能访问自己的数据）

### 范围外

- 社交功能（排行榜、好友）
- 全局段位赛匹配
- 数据看板 / 分析后台
- 自定义头像上传
- 邮箱验证 / 密码重置（Magic Link 不需要）

## 2. 整体架构

```
┌──────────────────────────────────────────────────┐
│                前端 (React + Zustand)              │
│                                                    │
│   Zustand Stores ── 接口不变 ──→ Repository        │
│                                   │                │
│                          ┌────────┴────────┐       │
│                          │   Repository    │       │
│                          │  (改造后)        │       │
│                          │                 │       │
│                          │  write(data):   │       │
│                          │   1. 写 local   │       │
│                          │   2. 通知 sync  │       │
│                          │                 │       │
│                          │  read(key):     │       │
│                          │   → 读 local    │       │
│                          └───┬─────────┬───┘       │
│                              │         │           │
│                    ┌─────────┘         └────────┐  │
│                    ▼                            ▼  │
│              localStorage              SyncEngine  │
│              (现有, 不变)               (新增)      │
│                                          │         │
│                                    ┌─────┴─────┐   │
│                                    │ AuthStore  │   │
│                                    │ (新增)     │   │
│                                    └─────┬─────┘   │
└──────────────────────────────────────────┼─────────┘
                                           │
                                    Supabase JS SDK
                                           │
                              ┌────────────┼────────────┐
                              │     Supabase Cloud      │
                              │  ┌──────┐ ┌──────────┐  │
                              │  │ Auth │ │PostgreSQL │  │
                              │  └──────┘ │  + RLS    │  │
                              │           └──────────┘  │
                              └─────────────────────────┘
```

### 核心设计原则

1. **Repository 接口不变**：Zustand stores 继续调用 `repository.saveGameProgress()` 等方法，不感知同步
2. **读操作永远读本地**：保证离线时零延迟
3. **写操作先落本地再推远端**：保证离线时正常使用
4. **SyncEngine 独立运行**：负责检测连接状态、推送本地变更、拉取远端变更、执行合并
5. **未登录 = SyncEngine 休眠**：访客模式零开销

## 3. Supabase 数据库设计

### 3.1 表结构

#### `profiles` — 用户资料

```sql
create table profiles (
  id uuid primary key references auth.users(id) on delete cascade,
  nickname text not null default '',
  avatar_seed text not null default '',
  settings jsonb not null default '{"soundEnabled": true, "hapticsEnabled": true}',
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now()
);
```

#### `game_progress` — 游戏进度（核心表）

```sql
create table game_progress (
  user_id uuid primary key references profiles(id) on delete cascade,
  campaign_progress jsonb not null default '{}',
  advance_progress jsonb not null default '{}',
  rank_progress jsonb not null default '{"currentTier": "apprentice", "history": []}',
  wrong_questions jsonb not null default '[]',
  total_questions_attempted int not null default 0,
  total_questions_correct int not null default 0,
  updated_at timestamptz not null default now()
);
```

> 设计决策：`campaign_progress` / `advance_progress` / `rank_progress` 保持 JSONB 而非拆子表。原因：
> - 这些数据的读写粒度是"整个 GameProgress 一起读、一起写"，和当前 localStorage 模式一致
> - 拆子表会让合并逻辑分散到多张表的事务中，增加复杂度但无实际收益
> - JSONB 配合 PostgreSQL 的 `||` 操作符可以做字段级 partial update

#### `history_records` — 历史答题记录

```sql
create table history_records (
  id text primary key,  -- 复用 PracticeSession.id (nanoid)
  user_id uuid not null references profiles(id) on delete cascade,
  session_mode text not null,
  started_at bigint not null,
  ended_at bigint,
  completed boolean not null default false,
  result text not null,  -- 'win' | 'lose' | 'incomplete'
  topic_id text not null,
  rank_match_meta jsonb,
  questions jsonb not null default '[]',
  synced_at timestamptz not null default now()
);

create index idx_history_user on history_records(user_id);
```

#### `rank_match_sessions` — 段位赛会话

```sql
create table rank_match_sessions (
  id text primary key,  -- nanoid
  user_id uuid not null references profiles(id) on delete cascade,
  target_tier text not null,
  best_of int not null,
  wins_to_advance int not null,
  games jsonb not null default '[]',
  status text not null default 'active',
  outcome text,
  started_at bigint not null,
  suspended_at bigint,
  cancelled_at bigint,
  ended_at bigint,
  updated_at timestamptz not null default now()
);

create index idx_rank_match_user on rank_match_sessions(user_id);
```

#### `sync_metadata` — 同步元数据

```sql
create table sync_metadata (
  user_id uuid primary key references profiles(id) on delete cascade,
  last_synced_at timestamptz not null default now(),
  device_id text
);
```

### 3.2 Row Level Security

所有表启用 RLS，策略统一：用户只能读写自己的数据。

```sql
-- profiles
alter table profiles enable row level security;
create policy "用户读写自己的资料" on profiles
  for all to authenticated
  using ((select auth.uid()) = id)
  with check ((select auth.uid()) = id);

-- game_progress
alter table game_progress enable row level security;
create policy "用户读写自己的进度" on game_progress
  for all to authenticated
  using ((select auth.uid()) = user_id)
  with check ((select auth.uid()) = user_id);

-- history_records
alter table history_records enable row level security;
create policy "用户读写自己的历史" on history_records
  for all to authenticated
  using ((select auth.uid()) = user_id)
  with check ((select auth.uid()) = user_id);

-- rank_match_sessions
alter table rank_match_sessions enable row level security;
create policy "用户读写自己的段位赛" on rank_match_sessions
  for all to authenticated
  using ((select auth.uid()) = user_id)
  with check ((select auth.uid()) = user_id);

-- sync_metadata
alter table sync_metadata enable row level security;
create policy "用户读写自己的同步元数据" on sync_metadata
  for all to authenticated
  using ((select auth.uid()) = user_id)
  with check ((select auth.uid()) = user_id);
```

### 3.3 自动创建 `profiles` 行

Supabase Auth 创建用户时只写入 `auth.users`，业务表 `profiles` 需要通过数据库 trigger 自动创建：

```sql
create or replace function handle_new_user()
returns trigger as $$
begin
  insert into profiles (id)
  values (new.id);
  insert into game_progress (user_id)
  values (new.id);
  insert into sync_metadata (user_id)
  values (new.id);
  return new;
end;
$$ language plpgsql security definer;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute function handle_new_user();
```

> 用 `security definer` 是因为 trigger 在 auth schema 上触发，需要跨 schema 写入 public 表。

### 3.4 自动更新 `updated_at`

```sql
create or replace function update_updated_at()
returns trigger as $$
begin
  new.updated_at = now();
  return new;
end;
$$ language plpgsql;

-- 每张需要 updated_at 的表都挂这个 trigger
create trigger set_updated_at before update on profiles
  for each row execute function update_updated_at();
create trigger set_updated_at before update on game_progress
  for each row execute function update_updated_at();
create trigger set_updated_at before update on rank_match_sessions
  for each row execute function update_updated_at();
```

## 4. 认证流程

### 4.1 Auth Store（新增）

```typescript
// src/store/auth.ts
interface AuthState {
  supabaseUser: SupabaseUser | null;  // Supabase auth.users 记录
  isAuthenticated: boolean;           // 是否已登录
  isLoading: boolean;                 // 认证状态加载中
  authError: string | null;

  // Actions
  initialize(): Promise<void>;       // 启动时检查 session
  signInWithMagicLink(email: string): Promise<void>;
  signOut(): Promise<void>;
}
```

### 4.2 登录流程

```
用户点击"登录" → 输入邮箱 → 调用 signInWithOtp({ email })
  → Supabase 发送 Magic Link 邮件
  → 用户点击邮件链接
  → 浏览器打开应用 + 自动完成认证
  → AuthStore.supabaseUser 更新
  → 触发 SyncEngine.onLogin()
  → 首次登录？ → 进入合并引导流程（§7）
  → 非首次  → 立即拉取远端最新数据并合并
```

### 4.3 Session 持久化

Supabase JS SDK 自动将 session token 存入 localStorage（key: `sb-<project-ref>-auth-token`），页面刷新后自动恢复。`AuthStore.initialize()` 在应用启动时调用 `supabase.auth.getSession()` 检查。

### 4.4 登出流程

```
用户点击"登出" → supabase.auth.signOut()
  → SyncEngine.onLogout()（停止同步、清除远端缓存）
  → AuthStore.supabaseUser = null
  → 应用回到访客模式，继续使用本地数据
```

登出**不清除本地数据**。用户可以继续以访客身份使用，或登录另一个账号。

## 5. Repository 层改造

### 5.1 改造策略

当前 `repository` 是一个单例对象，所有方法直接操作 localStorage。改造后：

- **读操作不变**：始终读 localStorage
- **写操作加一步**：写完 localStorage 后，将变更推入 SyncEngine 的 pending queue

```typescript
// 改造前
saveGameProgress(progress: GameProgress) {
  localStorage.setItem(KEYS.gameProgress(), JSON.stringify(progress));
}

// 改造后
saveGameProgress(progress: GameProgress) {
  localStorage.setItem(KEYS.gameProgress(), JSON.stringify(progress));
  syncEngine.markDirty('game_progress');  // 通知 sync engine
}
```

### 5.2 新增 localStorage key

```typescript
const KEYS = {
  // ... 现有 key 不变 ...
  syncState: () => `${keyPrefix}sync_state`,   // SyncEngine 持久化状态
  authUserId: () => `${keyPrefix}auth_user_id`, // 当前绑定的 Supabase user id
}
```

### 5.3 `syncState` 结构

```typescript
interface SyncState {
  lastSyncedAt: string | null;  // ISO timestamp，上次成功同步的时间
  dirtyKeys: string[];          // 待推送的数据 key 列表
  deviceId: string;             // 本设备唯一标识（首次生成，nanoid）
}
```

## 6. SyncEngine 设计

### 6.1 职责

SyncEngine 是一个独立模块，不依赖 React 渲染周期。它负责：

1. **检测在线状态**：监听 `navigator.onLine` + `window.ononline/onoffline`
2. **脏数据推送**：本地写入后，如果在线且已登录，立即尝试推送到 Supabase
3. **定期拉取**：每 30 秒检查远端是否有更新（通过 `updated_at` 比较）
4. **全量同步**：登录时、从离线恢复在线时，执行一次完整的 pull-merge-push 循环
5. **Realtime 订阅**：登录后订阅 `game_progress` 表的变更，其他设备写入时实时收到通知并拉取

### 6.2 同步协议

#### Push（本地 → 远端）

```
1. 检查 dirtyKeys 列表
2. 对每个 dirty key:
   a. 读取本地数据
   b. 转换为 Supabase 行格式
   c. upsert 到对应表
   d. 成功 → 从 dirtyKeys 移除
   e. 失败 → 保留在 dirtyKeys，下次重试
3. 更新 lastSyncedAt
```

#### Pull（远端 → 本地）

```
1. 查询 Supabase: 各表 where updated_at > lastSyncedAt
2. 对每条远端变更:
   a. 读取本地对应数据
   b. 调用 merge 函数（§6.3）合并
   c. 将合并结果写入本地
   d. 将合并结果推回远端（确保双向一致）
3. 更新 lastSyncedAt
```

#### 完整同步循环

```
Pull → Merge → Write Local → Push Merged → Update lastSyncedAt
```

### 6.3 合并策略（按数据类型）

#### GameProgress 合并

每个字段独立合并，规则基于数据的自然语义：

| 字段 | 合并规则 | 理由 |
|------|---------|------|
| `campaignProgress[topicId].completedLevels` | **集合并集**（按 levelId 去重，bestHearts 取 max） | 进度只增不减 |
| `campaignProgress[topicId].campaignCompleted` | **OR**（任一为 true 即 true） | 完成状态不可逆 |
| `advanceProgress[topicId].heartsAccumulated` | **取 max** | 单调递增 |
| `advanceProgress[topicId].sessionsPlayed` | **取 max** | 单调递增 |
| `advanceProgress[topicId].sessionsWhite` | **取 max** | 单调递增 |
| `rankProgress.currentTier` | **取最高段位** | 段位只升不降 |
| `rankProgress.history` | **按 startedAt 去重合并** | 追加制 |
| `rankProgress.activeSessionId` | **取非空值**；双方均非空时取远端（以最后联网的设备为准） | 联网检查保证不会两端同时活跃 |
| `wrongQuestions` | **按 (question.id + wrongAt) 去重合并**，合并后按 wrongAt 降序，截取前 100 条 | 追加制 + 上限 |
| `totalQuestionsAttempted` | **取 max** | 单调递增 |
| `totalQuestionsCorrect` | **取 max** | 单调递增 |

#### History Records 合并

- 按 `id`（= PracticeSession.id, nanoid）去重
- 远端有而本地没有 → 追加到本地
- 本地有而远端没有 → 推送到远端
- 两边都有且 id 相同 → 跳过（不可变数据，内容一致）

#### Rank Match Sessions 合并

- 按 `id` 去重
- 状态优先级：`completed` > `cancelled` > `suspended` > `active`
- 同一 session 两端状态不同时，取优先级更高的（终态不可逆）
- `games` 数组：取长度更长的版本（更多场次 = 更新的进度）

### 6.4 段位赛联网要求

段位赛（rank-match）要求**必须联网才能启动**（开始新系列 / 续打系列内下一局）。这是一个产品级约束，用来替代复杂的设备锁机制：

- **启动时检查**：`startRankMatch()` / `startRankMatchGame()` 先检查 `navigator.onLine` + Supabase 连接，离线时拒绝并提示"段位赛需要联网"
- **局中断网**：当前局正常继续（本地优先），完成后结果缓存到 dirtyKeys，联网后自动同步
- **多设备保护**：启动时从远端拉取最新 `rankProgress.activeSessionId`，如果远端已有活跃系列且非本设备发起，提示"请先在另一台设备完成段位赛"
- **闯关 / 进阶不受影响**：这两个模式完全离线可用

## 7. 首次登录合并引导

### 场景

用户已有本地进度（访客模式玩了一段时间），现在首次创建账号或登录。

### 流程

```
登录成功
  ↓
检测本地是否有数据 (user + gameProgress 非空)
  ├─ 无本地数据 → 直接从远端拉取（新设备登录已有账号）
  ├─ 有本地数据，远端无数据（新账号）→ 直接上传本地数据到远端
  └─ 有本地数据，远端也有数据（换设备 + 本地有旧进度）
       ↓
     弹出合并引导对话框：
     "发现本地有练习进度，你的云端账号也有进度。
      是否将本地进度合并到云端？"
     
     [合并到云端]  →  执行 §6.3 合并策略 → 结果写入双端
     [放弃本地]    →  用远端数据覆盖本地
```

### 合并后处理

- 将本地 `mq_auth_user_id` 设为当前 Supabase user id（UUID）
- **用户 ID 迁移**：本地 `User.id`（nanoid）替换为 Supabase UUID，`GameProgress.userId` 同步更新。旧 nanoid 不再使用。新创建的 PracticeSession / HistoryRecord 统一使用 Supabase UUID 作为 userId
- 标记合并完成，后续登录不再触发引导

## 12. 错误处理

| 场景 | 处理 |
|------|------|
| Magic Link 发送失败 | 显示"发送失败，请检查邮箱地址"，允许重试 |
| Magic Link 过期 | 提示"链接已过期，请重新发送" |
| 网络断开时写入 | 正常写入本地，dirtyKeys 记录待推送项 |
| 网络恢复后推送失败 | 指数退避重试（1s → 2s → 4s → 8s → 16s → 30s），6 次失败后标记 error 状态；Phase 3 开发文档以该口径为准 |
| 合并冲突（不应发生） | 所有合并规则确定性，不存在真正的冲突。如遇代码 bug 导致合并函数抛错，保留双方数据不覆盖，标记 error 让用户联系支持 |
| Supabase 服务不可用 | 等同离线处理，本地正常使用 |
| 段位赛离线启动 | 拒绝启动，提示"段位赛需要联网才能开始" |
| 段位赛多设备冲突 | 提示"请先在另一台设备完成段位赛" |
| localStorage 已满 | 现有问题，v0.3 不新增；如未来需要可考虑 IndexedDB 迁移 |

## 13. 测试策略

### 单元测试

- **合并函数**（最关键）：每种合并规则独立测试，覆盖各种组合
  - 本地空 + 远端有数据
  - 本地有 + 远端空
  - 双方都有、字段各有领先
  - 边界：wrongQuestions 超 100 截断、段位相同、history 完全重叠
- **SyncEngine 状态机**：dirty 标记、push/pull 流转、重试逻辑
- **Repository 改造**：确保写操作同时落本地 + 标记 dirty
- **版本迁移 v3→v4**：升级后数据完整性

### 集成测试

- 模拟离线/在线切换的同步行为
- 首次登录合并流程（三种场景）
- 段位赛联网检查（离线拒绝启动、在线正常启动）

### 手动验收

- 两台设备分别离线做题，联网后进度正确合并
- Magic Link 邮件收发 + 点击登录
- 同步状态指示器在各状态间正确切换
