# 执行纪律：每个子项的 4 步工作流

> 所属版本：v0.2  
> 所属主线：[README](./README.md)

---

## 背景

用户对工作方式的明确要求（2026-04-20）：

> "Claude 负责根据我提出的要求，提炼出相对当前版本优化后的预期效果，然后逐个问题根据预期的优化效果调研所需资料后，给出优化的方案由我审核。"

由此固化为下述 4 步通用流程。

## 四步通用流程

本主线下每个子项（无论属于 Phase 1/2/3/4/5），开工前统一按下列 4 步推进：

### 1. 预期效果提炼

**产出**：一段文字，明确说明"相对当前版本，优化后用户**可感知**的变化"。

**原则**：

- 从用户视角写，不写实现细节
- 避免"优化 XX"这种含糊词；用"优化后用户能做到 XXX / 看到 XXX / 不会再遇到 XXX"替代
- 覆盖"改好之后如何验证"的视角

### 2. 资料调研

基于预期效果，调研：

- 相关代码：生成器 / 页面 / Store / Spec
- 相关文档：`ProjectManager/Specs/` 对应规格
- 相关历史：`Reports/` · `ISSUE_LIST.md`
- 必要时：运行一次 Playwright 诊断、读 `reference-bank/`、比对现网截图

**产出**：一段事实清单，不做判断，只列"当前是什么状态"。

### 3. 方案设计

基于预期效果 + 资料调研，产出方案：

- 设计点
- 影响面（代码 / 文档 / 数据 / 回归范围）
- 风险与缓解
- 工作量估算（小时级或人·天级）

### 4. 审核

把方案提交给用户审核：

- 通过 → 进入实施，按 Plan 模板落盘 `Plan/v0.2/subplans/YYYY-MM-DD-<feature-slug>.md`
- 不通过 → 回到步骤 1，按反馈重做

## 为什么这样做

- 用户显式要求：Claude 负责"预期提炼 → 调研 → 方案"，用户负责"审核"
- 避免直接跳到实施后返工
- 每一步都有明确产出，中断/恢复成本低（符合项目"会话交接纪律"）

## 什么时候可以跳过

- **纯执行性小改动**：例如按已审批规格一比一落代码
- **诊断单本身**：诊断单即"调研 + 结论"，不需要再走一遍预期提炼
- **修正上一轮审核意见**：可直接回到步骤 3 修正方案，不必重写预期效果

## 每步的产出形式建议

| 步骤 | 建议形式 | 长度 |
|---|---|---|
| 1. 预期效果 | 对话中一段文字或 Plan 头部段落 | 200~500 字 |
| 2. 资料调研 | 对话中的事实清单 / 报告表格 | 根据复杂度 |
| 3. 方案设计 | 对话中结构化方案 / 若已成熟则起草 Plan 文件 | 根据复杂度 |
| 4. 审核通过 | Plan 文件正式落盘 `ProjectManager/Plan/v0.2/subplans/*.md` | 按 Plan 模板 |
