## 范围决策

v0.5 纳入以下事项：

- `BL-009`（闯关竖式题目需排除纯口算样例）。
- `BL-011`（计算输入内置键盘与移动端默认键盘策略）。
- `BL-010`（竖式除法 UI 化答题功能）。
- `ISSUE-067`（多行乘法竖式判错面板缺少过程 / 训练格错因）。

`BL-012`（特定账号数据查询后台页面）更适合拆分到 v0.6。原因是它属于账号 / Supabase / 后台查询系统维度，风险模型和学生端 A03 / Practice 输入体验不同；放进同一个 release gate 会扩大 v0.5 的范围和验收成本。

## BL-009 预研结论：闯关竖式题排除纯口算样例

- 用户反馈样例：`888 ÷ 4`、`844 ÷ 4`、`208 ÷ 4`、`672 ÷ 4`、`2位数 × 1位数` 等题目不需要通过竖式完成。
- 已确认的产品原则：普通五年级学生大概率能心算的题，不适合作为竖式训练题。
- A/B 关系结论：A 是“竖式题不应出纯口算题”的设计逻辑；B 是产品里可执行的具体过滤规则。B 是 A 的落地子集。
- v0.5 Phase 策略：先抽样诊断，再根据诊断结论判断强制过滤规则；过滤必做，但规则不在诊断前拍死。
- 详细设计需回答：抽样范围、口算样例判定口径、过滤发生在生成器还是 campaign lane、是否影响已有进阶 / 段位赛样本池。

## BL-010 预研结论：竖式除法 UI 化答题

### 总目标

- 模拟真实除法竖式答题过程，像 v0.4 乘法竖式 UI 一样训练过程能力。
- 训练与准确率、效率相关的关键能力：商位判断、试商 / 调商、乘积、相减、余数小于除数、落位 / 补 0、商中间 0、小数点规则、取近似、循环节识别。
- 上海小学五年级会遇到的相关题型都要覆盖。

### UI 架构结论

- 采用“一套动态整数长除法核心板 + 外围扩展训练格”。
- 删除短除法方向，只保留长除法。短除法与心算差别不大，不适合作为本次 UI 化重点。
- 长除法核心板按用户填写过程动态扩展：当前轮填商位、乘积、余数；系统落下一位或补 0；再进入下一轮。
- 同一套核心机制兼容除数为 1、2、3 位的情况；详细设计阶段再确认最大位数和移动端宽度限制。

### 小数 / 取近似 / 循环小数结论

- 小数 ÷ 整数：商的小数点与被除数小数点对齐，系统预置小数点，不让学生额外填写。该环节简单、低错率，拿掉后不明显损失训练效果。
- 小数 ÷ 小数：计算前提供扩倍训练格，例如“扩倍倍数”“扩大后的被除数”“转换后的整数除法”。
- 取近似：计算前可提示“保留 X 位，需要算到第 X+1 位”；计算后只填写“保留 X 位小数结果”，不再让学生填写“第 X+1 位小数是多少”。
- 循环小数：循环节识别需要填；最终答案不让学生输入循环点，改为结构化输入“非循环部分 + 循环节”，由系统展示循环小数写法。
- 循环小数屏幕适配规则：限制循环节最长 3 位、非循环部分最多 2 位小数、展示字符串总长不超过 8 个字符；超限题不进入循环节识别题池，可转入取近似题池。
- 循环小数训练格不再使用“第一次重复出现的余数”“已经看到重复的商数字”这类字段；只保留学生需要识别和填写的“非循环部分 + 循环节”。

### 判定结论

- 竖式除法采用严格过程判定：所有需填写的长除法过程格和扩展训练格都必须正确才通过。
- 难度改变题目复杂度和训练格范围，不改变通过条件。
- 系统预置的小数点不作为判定格。

## BL-011 预研结论：计算输入内置键盘

- 目标：内置键盘体验要好于系统键盘，避免移动端每题手动切数字模式、避免系统键盘遮挡题干、降低页面适配负担。
- 接入范围：全量接入所有计算字符输入入口，包括普通答案、商余数、多空填空、表达式 / 等式、竖式格、多行乘法、小数训练格、未来长除法格；选择题 / 判断题不接入。
- 键盘档位：当前题型两档足够。
- 基础数字键盘：`0-9`、`.`、`-`、删除、清空、上一格、下一格、确认、系统键盘。
- 表达式扩展栏：`+`、`-`、`×`、`÷`、`(`、`)`、`=`、`x`。
- Chrome / Safari 要稳定满足“移动端默认内置键盘、全量接入字符输入”。
- 其他浏览器允许降级为“内置键盘优先 + 系统键盘可同时弹出”。
- 稳定技术方向：默认答题不聚焦真实 `input` / `textarea` / `contenteditable`，使用自绘输入格和内置键盘写入状态；`inputmode` 可作为辅助，不把 `virtualKeyboardPolicy` 作为核心依赖。
- 系统键盘兜底入口不能只放在内置键盘上。答题区或当前输入格附近必须有独立可见入口，避免内置键盘异常时无法呼出系统键盘。

## ISSUE-067 预研结论：多行乘法竖式判错面板错因

- 现象：多行乘法竖式中，过程格或小数乘法训练格错误、最终答案正确时，判错面板只显示正确答案，不显示错因。
- 当前核查结论：多行乘法竖式提交时过程格 / 训练格必须正确；最终答案正确但过程错误仍不通过。
- v0.5 修复目标：最终失败面板必须说明本题未通过的原因，避免用户误以为答案正确却被系统误判。
- 计算过程格错误：按出错性质类别显示，不显示具体错误值和正确值。例如“商位错误”“乘积错误”“相减 / 余数错误”“落位 / 补 0 错误”“竖式过程填写错误”。
- 扩展训练格错误：必须写清具体类型，并显示用户值和正确值。例如“扩倍转换错误”“整数除法转换错误”“取近似结果错误”“非循环部分错误”“循环节错误”。
- 历史记录 UI 不显示错误原因；错题本需要显示错误原因。
- `HistoryRecord` 底层可以保留 `failureReason`，作为未来分析数据来源。

## BL-012 拆分结论：特定账号数据查询后台页面

- 结论：拆分到 v0.6，不进入 v0.5。
- 原因：该需求牵涉 Supabase 数据、权限、隐私、只读保护、后台入口和运维流程，属于账号 / 在线服务维度；与 v0.5 的 A03 / Practice 输入体验主线不同。
- v0.6 详细设计必须确认：查询入口、权限与访问控制、可查询账号定位方式、Supabase 表范围、是否只读、隐私边界、审计记录、生产环境可见性。
- 默认低风险方向：只读查询后台，服务排障和数据核查，不提供修改能力。
