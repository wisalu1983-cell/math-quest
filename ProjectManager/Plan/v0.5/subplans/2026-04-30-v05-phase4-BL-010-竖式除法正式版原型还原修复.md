## 文档一致性核查

本轮修复前核查结果：现有 `BL-010` 主开发文档与 formal prototype 存在三处表达缺口。

| 维度 | formal prototype 当前设计结果 | 现有文档状态 | 处理 |
|---|---|---|---|
| 小数 ÷ 小数阶段流 | 扩倍确认通过后，转换区退出页面主体，只显示竖式计算和提交 | 主文档只写“通过后进入竖式板”，未显式写“转换区不再作为主输入区显示”；上一份原型 bugfix 文档还写“本轮不修生产组件” | 本文补为正式组件修复约束 |
| 竖式板形态 | 纸面长除法：弯除号、横线、被除数逐位排布、商位在上方、乘减结果按位对齐 | 主文档引用审核稿，但未把正式还原差异写成 bugfix 范围 | 本文列入生产 UI 修复范围 |
| 输入交互 | 逐位小格 + 当前格焦点；初始只显示当前可填格，随输入顺序逐步显现后续过程格 | 主文档写“动态逐轮展开”，但生产组件实际一次铺满所有大输入框 | 本文明确以 formal prototype 的逐步显现作为修复目标 |
| 错误输出 | 过程格错误按具体步骤显示，例如 `第 1 轮商位错误`；扩倍、取近似、循环节等结构化字段显示用户值和正确值 | 主文档仍保留通用类别示例，生产判定也沿用 `商位判断错误` 等通用文案 | 本文补为正式修复范围并由 parity QA 覆盖 |
| 循环小数表达 | 商位末尾显示省略号；结果区显示非循环部分、循环节和带循环点的标准格式答数 | 生产只显示普通循环小数文本，未还原原型的标准格式预览 | 本文补为正式修复范围并由 parity QA 覆盖 |

结论：开发文档需要先补充正式版原型还原 bugfix 范围，再进行生产代码修改。

## 问题现象

当前正式组件 `src/components/LongDivisionBoard.tsx` 与 formal prototype 差距明显：

- 长除法过程格使用整字段大输入框，例如 `13` 是一个宽输入框，而 prototype 是两个逐位小格。
- 所有商位、乘积、余数 / 落位格一次性铺满，缺少 prototype 的逐步显现。
- 小数 ÷ 小数扩倍确认通过后，转换区仍留在竖式板上方。
- 竖式板使用普通 CSS grid 框体，未还原 prototype 的纸面长除法版式。
- 过程错误反馈仍使用通用类别，未按 prototype 显示具体步骤错误。
- 循环小数缺少商位省略号和标准格式答数循环点预览。

## 复现路径

1. 在正式 Practice 中注入或生成带 `data.longDivisionBoard` 的 A03 除法题。
2. 观察整数长除法 `936 ÷ 4`：页面一次显示 9 个过程输入框。
3. 观察小数 ÷ 小数：扩倍确认后仍显示转换区。
4. 与 `/?preview=longdiv-formal` 的同类题型对照，差异集中在板面、输入粒度、显现顺序和阶段替换。

## 用户影响

- 正式版本没有呈现“真的在列竖式”的纸面心智。
- 一次铺满所有格子会增加低龄学生的视觉负担，弱化“商、乘、减、比、落”的步骤感。
- 小数 ÷ 小数扩倍区不退出，会让学生误以为扩倍字段仍是当前阶段的主任务。
- 移动端高度被无关阶段占用，增加键盘遮挡和滚动负担。

## 根因

生产组件在 Phase 4 落地时只承接了数据结构、判定和键盘 slot，未把审核预览 / formal prototype 的 UI 细节沉淀到生产组件：

- `renderInput()` 以字段为单位渲染单个 input。
- `renderQuotientInputs()` 与 `renderRoundRows()` 遍历所有轮次，无逐步显现条件。
- `renderSetupPanel()` 在 root return 中无条件渲染，只由 `data.setupFields` 控制。
- 纸面长除法所需的逐位定位、弯除号、商 / 被除数小数点、横线和自适应布局仍停留在 preview 组件内。
- 过程错因分类沿用早期通用 label，没有用长除法字段 label 生成与 prototype 一致的步骤级类别。
- 扩倍字段完成条件误用了“达到标准答案长度”，导致错误值 `10` 不能触发 `扩倍倍数填写有误`。

## 修复范围

### 纳入

- 将生产 `LongDivisionBoard` 的核心竖式板改为 formal prototype 的纸面版式。
- 核心过程字段改为逐位小格输入，同时保持对外 `values[fieldId]` 为完整字段字符串，保证判定函数不改。
- 按当前 active slot / 已填写值 / submitted 状态逐步显现过程格；初始只显示第一轮商位。
- 小数 ÷ 小数扩倍确认通过后隐藏转换区，主体进入竖式计算。
- 小数 ÷ 小数扩倍错误值可提交校验，中档停留转换区并按字段显示 `扩倍倍数填写有误` / `转换后除数填写有误` / `转换后被除数填写有误`。
- 循环小数还原商位省略号和结果区“标准格式答数”循环点预览。
- 长除法过程错误反馈按具体步骤显示，例如 `第 2 轮余数与落位错误`，不泄露中间正确值。
- 保留 Phase 3 内置键盘 slot 协议：`enabledKeys`、`sanitizeInput`、`shouldAutoAdvance`、active slot、桌面 Tab 顺序。
- 保留结构化错因和提交 payload 形状；`classifyLongDivisionSubmission()` 的过程错误 label 对齐 formal prototype。
- 补生产 E2E 回归，覆盖逐步显现、逐位输入、小数 ÷ 小数阶段替换。

### 非目标

- 不改变题目生成器和长除法标准过程生成。
- 不改变失败原因枚举或错题本展示口径。
- 不把审核预览 `/?preview=longdiv` 的说明面板移入正式组件。
- 不新增短除法 UI。

## 选定方案

选定方案：在生产 `LongDivisionBoard` 内复刻 formal prototype 的核心板渲染与逐位输入模型，但继续复用生产数据、判定和键盘协议。

理由：

- 风险低于直接依赖 preview 组件；preview 含审核控制、场景切换和演示数据，不适合作为生产组件。
- 可以保持 `LongDivisionBoardData`、`VerticalCalcCompletePayload`、`PracticeMathKeyboard` 不变。
- 逐位输入只改变 UI 层；字段值仍聚合为完整字符串，判定函数可原样复用。

未采用方案：把 preview 组件抽成共享生产组件。

- 放弃原因：preview 组件含大量审核态、场景数据、演示按钮和说明文案，抽共享会扩大改动面并增加后续维护风险。

## 实现约束

- 前置扩倍字段和后置结构化结果字段继续使用 field-level `MathInputSlot`。
- 核心过程字段在 UI 和键盘层拆为 digit-level `MathInputSlot`；digit input 的 `aria-label` 使用 `第 N 轮商位第 M 位` 等名称。
- 数字键写入 active digit 后，更新完整字段值；digit slot 按顺序自动进入下一格，判定函数仍读取原 `values[fieldId]` 完整字符串。
- 已填写字段、当前字段、submitted 后的字段可见；未来字段不可见。
- 转换区只在 `!conversionConfirmed` 时作为主输入区显示；确认后不再渲染主转换区。
- 小数点由系统显示，不作为输入格；商小数点与被除数小数点沿 `quotientDecimalAfter` 和工作被除数显示。
- 移动端不能产生全页横向滚动；长循环样例允许板内压缩或局部滚动，但当前修复至少不得劣化既有 375 / 390 证据。
