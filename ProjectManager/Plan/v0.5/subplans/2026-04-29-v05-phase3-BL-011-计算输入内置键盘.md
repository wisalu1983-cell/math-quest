## 一、背景

`BL-011` 要解决移动端计算题输入体验：系统键盘会遮挡题干、每题切换数字 / 符号模式成本高，且不同浏览器行为不稳定。v0.5 预研已确认首版采用内置计算键盘，移动端默认不弹系统键盘；系统键盘呼出能力在工程层保留，但首版 UI 屏蔽，不展示碍眼按钮。

本计划从现有 `Practice` 输入状态、`VerticalCalcBoard` 单一字符入口、`MultiplicationVerticalBoard` 多输入格和未来长除法格出发，建立统一计算输入协议。实现后，题型组件只声明“有哪些可输入槽位、允许哪些字符、如何提交”，键盘只负责字符写入和删除；移动槽位通过直接点选 UI 完成，确认仍走各题型答题区既有按钮。

### 工作脉络

1. v0.4 已完成 Practice 输入态 reset 和 legacy 单行竖式单一字符入口，说明输入状态可以收敛到页面级 hook 与板内单一入口。
2. v0.5 Phase 1 已确认内置键盘归属 Practice 输入基础设施，题型组件只声明输入格与动作。
3. Phase 3 先做输入槽位协议和键盘 UI，再逐步接入现有计算输入，避免在 Phase 4 长除法里重新实现一套输入系统。
4. Phase 3 验收通过后，Phase 4 长除法只需要注册长除法过程格 / 训练格，不再决定移动端键盘策略。

## 二、目标与范围

### 目标

- 建立统一的计算输入槽位模型，支持插入和删除；槽位切换通过直接点选目标 UI，确认沿用题型原有答题按钮。
- 提供两档键盘：基础数字键盘与表达式扩展栏。
- 移动端默认不聚焦真实 `input/textarea/contenteditable`，避免自动弹系统键盘。
- 系统键盘呼出能力在工程层保留，但首版 UI 屏蔽，不在答题区或内置键盘上显示按钮。
- 内置键盘必须控制高度；任何手机视口下，答题区高度占整体屏幕高度不得低于 60%。
- 让现有 Practice 主要计算输入入口接入统一键盘。
- 保持桌面键盘输入可用，避免为移动端优化牺牲桌面效率。

### 非目标

- 不实现长除法 UI；只预留长除法输入槽位能力。
- 不重做选择题、多选题、判断题输入。
- 不改题目判定规则、心数、结算或段位赛规则。
- 不把输入态持久化到 session；换题和刷新后仍按当前题重建输入态。
- 不用实验性 API 保证核心行为。

## 三、功能逻辑

### 3.1 接入范围

| 输入形态 | 是否接入 | 说明 |
|---|---|---|
| 普通数字 / 小数答案 | 是 | `numeric-input` 的主答案，支持 `0-9`、`.`、`-`，按题目约束禁用不适用键。 |
| 除法商余数 | 是 | 商、余数各为独立槽位，学生直接点选要编辑的槽位。 |
| 多空填空 | 是 | 每个 blank 是独立槽位，按题目类型决定允许字符。 |
| 表达式 / 等式输入 | 是 | 主输入槽位支持扩展栏：`+ - × ÷ ( ) = x`。 |
| legacy 单行竖式格 | 是 | 复用既有单一字符入口，不恢复隐藏 input 双消费。 |
| 多行乘法竖式格 | 是 | 每个部分积 / 合计 / 小数训练格注册为槽位。 |
| 小数训练格 | 是 | 小数位数、扩倍、移动位数等训练格注册为槽位。 |
| 未来长除法格 | 预留 | Phase 4 使用同一槽位协议。 |
| 选择题 / 判断题 / 多选按钮 | 否 | 仍用按钮交互。 |

### 3.2 键盘模式

| 模式 | 按键 | 触发条件 |
|---|---|---|
| 基础数字键盘 | `0-9`、按槽位需要显示 `.` / 负号 `-`、删除 | 普通数字、小数、商余数、竖式格、训练格 |
| 表达式扩展栏 | `+`、`-`、`×`、`÷`、`(`、`)`、`=`、`x` | expression-input / equation-input |

规则：

- 数字键盘只展示当前槽位必要字符，不把不可用字符以 disabled 占位塞进 UI。
- 内置键盘不常驻 `确认` 键；提交动作仍由各题型答题区既有按钮承担，避免同屏出现两个确认入口。
- 内置键盘不常驻 `上一格` / `下一格`；多槽位题型通过手指直接点选目标格切换 active slot。
- `删除` 删除当前槽位光标前字符；单字符格直接清空。
- `清空` 与连续删除功能重叠，首版不常驻；如后续需要，可在长按删除或辅助菜单里评估。
- 表达式输入允许在光标位置插入；单字符格只接受最后一次有效输入。
- 负号 `-` 在基础数字键盘里表示 signed number 的负号，只在当前槽位允许负数时显示；表达式扩展栏里的 `-` 表示减法运算符。

### 3.3 移动端默认策略

- 默认内置键盘：触摸设备或移动视口下，点击计算输入槽位只激活槽位和显示内置键盘，不主动 focus 真实输入控件。
- 系统键盘兜底：保留当前槽位进入系统键盘模式的工程能力，首版 UI 不展示呼出按钮；测试可通过内部接口 / dev hook 验证能力存在。
- 桌面策略：桌面可保留真实输入焦点和硬键盘输入；内置键盘可作为辅助操作面板。
- 浏览器降级：Chrome Android / Safari iOS 必须满足默认内置键盘；其他浏览器若无法稳定阻止系统键盘，允许出现“内置键盘优先 + 系统键盘可同时弹出”的降级表现。

### 3.4 系统键盘 UI 策略

Phase 3 采用以下收束口径：

- 首版 UI 不展示系统键盘呼出按钮。
- 工程层保留 `useSystemKeyboardForSlotId` 或等价状态，后续可在设置项、辅助功能或长按槽位中开放。
- QA 只验证该能力未破坏默认内置键盘路径，不要求用户可见入口。

## 四、UI / 交互设计

### 4.1 页面结构

- 当前输入槽位使用明显 active 样式，避免学生不知道按键会写入哪里。
- 当前槽位可以有一个放大编辑区：它不是额外答案格，而是当前 active 槽位的值镜像，用于在手机上明确“键盘正在写入哪一格”。竖式格、多行乘法格等小格场景优先使用该编辑区。
- 内置键盘在 Practice 答题态显示，反馈态隐藏。
- 手机竖屏下键盘贴近底部安全区，题干、当前输入区和题型原确认按钮不被遮挡；答题区高度占整体屏幕高度不得低于 60%。
- 桌面宽度下键盘保持在答题卡下方或页面底部，不撑破 `max-w-lg` 主容器。

### 4.2 按钮设计

- 数字 / 符号键为固定尺寸网格按钮。
- 删除使用 `lucide-react` 图标 + tooltip / aria-label；必要时保留短文字标签。
- 不可用字符不展示；只有短暂加载或权限态才使用 disabled 状态。
- 触摸目标不小于 44px 高。

### 4.3 可访问性

- 当前槽位有 `aria-label`，说明题目内语义，例如“商”“余数”“第 1 个部分积第 3 格”。
- 键盘按钮有 `aria-label`，删除不只依赖图标。
- 系统键盘切换能力如果通过内部入口触发，焦点进入当前真实输入控件；退出系统键盘模式后回到内置键盘槽位状态。

### 4.4 异常与边界

| 场景 | 处理 |
|---|---|
| 没有 active 槽位时按键 | 默认激活第一个可输入槽位；若没有槽位，键盘不显示。 |
| 当前槽位已满时继续输入 | 单字符格替换当前值；多字符格按 maxLength 截断。 |
| 输入非法字符 | 忽略并保留原值；不震动、不清空。 |
| 系统键盘模式输入非法字符 | 走同一 sanitize 规则。 |
| 切题 / 下一题 | 输入态按新题重建，键盘 active 槽位指向首个可输入槽位。 |
| 提交后反馈态 | 键盘隐藏；Enter / 题型确认按钮不重复提交。 |
| 内置键盘异常不可见 | 首版 UI 不提供可见系统键盘按钮；按功能故障处理，进入错误兜底或刷新恢复，不在界面上额外放按钮。 |

## 五、工程实现方案

### 5.1 新增核心类型

建议在 Practice 同级新增输入基础设施文件，避免把 `Practice.tsx` 继续膨胀：

- `src/pages/practice-math-keyboard.ts`：纯 reducer、类型、按键策略、slot 操作。
- `src/pages/PracticeMathKeyboard.tsx`：键盘 UI。
- `src/pages/practice-math-keyboard.test.ts`：纯逻辑测试。

核心类型建议：

```ts
export type MathKeyboardMode = 'numeric' | 'expression';

export interface MathInputSlot {
  id: string;
  label: string;
  value: string;
  maxLength?: number;
  mode: MathKeyboardMode;
  allowDecimal?: boolean;
  allowNegative?: boolean;
  allowedChars?: string[];
  setValue: (next: string) => void;
  focusSystemInput?: () => void;
}

export interface MathKeyboardState {
  activeSlotId: string | null;
  useSystemKeyboardForSlotId: string | null;
}
```

实现时可按实际代码微调命名，但必须保留三个契约：slot id 稳定、写入通过 slot setter、系统键盘切换能力只作用于当前槽位且首版 UI 屏蔽。

### 5.2 与现有 `usePracticeInputState()` 的关系

- `usePracticeInputState()` 继续持有题目输入值：`answer`、`remainderInput`、`blankValues`、`trainingValues`。
- 内置键盘不另建并行答案状态，只通过 slot setter 写入现有 state。
- 换题 reset 仍由 `usePracticeInputState(currentQuestion)` 负责。
- 焦点副作用从“默认 focus input”调整为“移动端默认激活首槽位；桌面可 focus 输入框”。

### 5.3 题型接入

| 文件 | 改动 |
|---|---|
| `src/pages/Practice.tsx` | 根据当前题注册 slots；渲染 `PracticeMathKeyboard`；提交态读取原有 state。 |
| `src/pages/practice-input-state.ts` | 增加必要的 slot setter / cursor 辅助；保持 reducer 纯度。 |
| `src/components/VerticalCalcBoard.tsx` | 暴露或接收键盘输入适配；保持单一字符消费入口。 |
| `src/components/MultiplicationVerticalBoard.tsx` | 输入格改为可被 slot 写入；同时保留桌面真实输入路径。 |
| 未来 `LongDivisionBoard` | Phase 4 直接注册长除法槽位，不重新设计键盘。 |

### 5.4 不采用的方案

| 方案 | 放弃原因 |
|---|---|
| 只设置 `inputmode` / `type=number` | 不能保证 Safari / Chrome 默认不弹系统键盘，也不能解决符号键切换成本。 |
| 依赖 `virtualKeyboardPolicy` | 兼容性不足，不能作为 Safari 核心方案。 |
| 每个题型各自实现一个键盘 | 会复制系统键盘策略、按键规则、移动端布局和 QA 成本。 |
| 键盘独立持有答案状态 | 容易与 `Practice` 提交状态分叉，增加恢复和 reset 风险。 |

### 5.5 实现架构决策门

| 维度 | 结论 |
|---|---|
| 模块 ownership 与依赖方向 | Practice 输入基础设施 owns 键盘 reducer、键盘 UI、slot 注册；题型组件 owns 自己的输入槽位定义；Store 不持有临时键盘状态。 |
| 类型与 API 契约 | 新增 `MathInputSlot` / `MathKeyboardState` / 键盘 action；输入值仍写回 `usePracticeInputState()` 或题型组件现有 state。 |
| 持久化 / 存档 / 版本迁移 | 不持久化键盘 active slot、系统键盘模式或临时输入光标；无存档版本 bump。 |
| 标识符命名空间与向后兼容 | slot id 使用 `answer-*`、`blank-*`、`vertical-*`、`mult-*` 前缀；不改题目 id、session id 或历史答案格式。 |
| UI 容量与响应式约束 | 375px 竖屏必须验证键盘高度、答题区高度占比、当前输入格和题型原确认按钮；答题区高度占整体屏幕高度不得低于 60%；首版无可见系统键盘入口。 |
| 测试与 QA 映射 | reducer 单测覆盖输入 action；Practice / 竖式输入用 Playwright 或拟真 QA 覆盖；Chrome Android / Safari iOS 是强制验收路径。 |
