### 4.1 长除法轮次模型

Q1 已有前置预研结论，不重新作为开放问题处理。

来源：

- [`2026-04-28-v05-preresearch-raw-dialogue.md`](../2026-04-28-v05-preresearch-raw-dialogue.md)：用户在过程填写粒度问题中选择“全过程填写”；随后记录为每轮填写“当前商位、商 × 除数的乘积、相减结果、落下下一位后形成的新工作数”。
- [`2026-04-28-v05-phase1-开工对齐与跨phase边界.md`](./2026-04-28-v05-phase1-开工对齐与跨phase边界.md)：P4 已确认“让学生自己填落位 / 补 0；系统负责引导、校验和错误反馈”。
- [`../phases/phase-4.md`](../phases/phase-4.md)：Phase 4 前置状态再次列明“学生自己填写落位 / 补 0”。

采用结论：

```text
每一轮长除法过程：
1. 系统展示当前被除数段 currentPartialDividend
2. 学生填写当前商位 quotientDigit
3. 学生填写商位 × 除数的乘积 product
4. 学生填写相减结果 / 当前余数 remainder
5. 系统执行“比”：校验当前余数是否小于除数，不新增学生输入格
6. 学生填写落位 / 补 0 后形成的新工作数 nextPartialDividend
7. 系统校验本轮并推进下一轮
```

边界解释：

- “动态逐轮展开”仍成立：UI 不预先铺满整张长除法表，而是在每轮校验后追加下一轮。
- “系统落下一位 / 补 0”的预研简写只表示系统推进流程、展示来源数字并打开下一轮，不表示系统代填学生应训练的 `nextPartialDividend`。
- “比”是教学顺序中的检查动作，不是竖式书写结果；实现上作为系统校验 / 提示节点，不放入 `MathInputSlot`。
- 最后一轮如果已经到达终止条件，不再要求填写下一轮 `nextPartialDividend`；若进入继续除、取近似或循环小数识别，则补 0 后的新工作数仍是判定格。
- 过程格错因必须能区分：商位 / 试商错误、乘积错误、相减 / 余数错误、余数大小判断错误、落位 / 补 0 错误；“比”触发的问题按 Q5 的 `long-division-compare` 分类。

#### 4.2.1 UI 任务映射

| 题型来源 | UI 任务 | 长除法核心板计算对象 | 扩展训练格 / 后置格 | 判定口径 |
|---|---|---|---|---|
| 整数除法：`int-div` 低档整数商；中档整数除出小数 | `long-division-core` | 原始 `dividend ÷ divisor` | 通常无前置训练格；若除不尽继续补 0，由核心板轮次处理 | 每轮商位、乘积、相减 / 余数、落位 / 补 0 后新工作数全正确才通过 |
| 小数 ÷ 整数：`dec-div` 中档 | `long-division-decimal-dividend` | 原始 `decimalDividend ÷ integerDivisor` | 商的小数点系统预置，不作为学生输入格；必要的补 0 继续除由核心板处理 | 长除法过程格全正确；系统预置小数点不参与判定 |
| 小数 ÷ 小数：`dec-div` 中高档 | `decimal-divisor-conversion + long-division-core` | 扩倍后的除法，例如 `15.6 ÷ 0.24 -> 1560 ÷ 24`，也允许 `1.234 ÷ 0.04 -> 123.4 ÷ 4` | 计算前转换区分成“扩倍确认”和“实际计算”两步：先连续填写扩倍倍数、转换后的除数、转换后的被除数，并点击“确认扩倍”；三个转换输入格小数点键始终可用，等值小数按数值归一比较；转换后除数必须可归一为整数，否则提示并阻止进入竖式板；非高档转换值填错只提示字段填写有误，不给正确值，也不进入竖式板；高档转换值填错直接判本题过程失败并展示具体错因；通过后按转换后被除数选择小数点模式 | 扩倍确认通过后才进入竖式板，并按通用逻辑判定长除法过程格；取近似 / 循环结构等字段错因仍进入反馈 |
| 取近似：`approximate` 中 `operation='÷'` | `long-division-approximation` | 原始除法或扩倍后的除法，计算到保留位后一位 | 计算前提示“需要算到第 X+1 位”；计算后只填写“保留 X 位小数结果” | 长除法过程格和最终近似结果都正确才通过；不额外填写第 X+1 位数字 |
| 循环小数：除法结果进入循环识别任务 | `long-division-cyclic + cyclic-result-fields` | 长除法动态追加商位，直到可识别循环或达到上限 | 后置结构化输入：完整非循环部分、循环节；系统按中国常见循环点格式展示标准答数，不使用括号格式 | 过程格、非循环部分、循环节都正确才通过 |

#### 4.2.2 现有代码路由边界

- `vertical-calc` 里的 `approximate` 目前可能生成乘法取近似；Phase 4 只接管其中 `operation='÷'` 的题目。`operation='×'` 的取近似仍走现有乘法 / 数值路径，不能误路由到 `LongDivisionBoard`。
- `decimal-ops` 里的 `cyclic-div` 属于 A05 小数运算题型；`BL-010` 的循环小数只覆盖进入 A03 竖式除法 UI 的任务，不自动接管 A05 全部循环小数题。
- `int-div` 中档已经可能生成“整数 ÷ 整数，商为有限小数”的题；它归入长除法核心板，不需要小数 ÷ 整数扩展训练格。

#### 4.2.3 循环小数上限

循环小数沿用预研确认的移动端适配规则：

- 循环节最长 3 位。
- 非循环部分填写完整前缀，包含整数部分和小数点；最多 2 位小数。例如 `14 ÷ 135` 的非循环部分填 `0.1`，不是只填 `1`。
- 展示字符串总长不超过 8 个字符。
- 超限题不进入循环节识别题池，可转入取近似题池。
- 竖式板必须让循环节至少完整出现 2 次后才开放后置循环结构结果格；这不改变“循环节最长 3 位”的结果字段上限，但会增加竖式板过程轮次，需纳入移动端容量校验。
- 标准格式答数由系统根据“非循环部分 + 循环节”生成；按中国小学考卷 / 练习册常见写法，在循环节 1 位时给该位上方打点，循环节多位时给首位和末位上方打点。

#### 4.2.4 不作为输入格的内容

- 小数 ÷ 整数的商小数点：系统预置，不要求学生填写。
- 整数 ÷ 整数的商小数点：默认不显示；若个位处理完仍有余数且本题要求继续算小数，则系统在商的个位后自动出现。
- 小数 ÷ 小数的扩倍确认：扩倍倍数、转换后除数、转换后被除数必须先提交并通过；转换后除数必须可归一为整数，若用户仍填写小数，先在转换区警告，不生成长除法核心板。
- 取近似的第 `X+1` 位数字：只作为长除法计算到位的内部过程和提示，不单独作为训练格。
- 循环小数的“第一次重复出现的余数”“已经看到重复的商数字”：删除，不进入 UI 字段；只保留“完整非循环部分 + 循环节”，标准格式答数由系统显示循环点。

### 4.3 输入与自动换格

Q3 用户确认需要一个默认输入顺序，并要求参考上海小学数学教学大纲 / 教材体系里的竖式除法计算规范。当前可访问的上海市教委公开材料能确认课程标准与教学用书入口，但未查到公开网页直接逐字列出“商、乘、减、比、落”的条款；本计划据此采用小学笔算除法通行教学顺序作为实现默认，并保留后续按上海教材具体页码精确校准的空间。

- 上海市教委公开入口：[`2026年上海市中小学教学用书目录`](https://edu.sh.gov.cn/xxgk2_zdgz_jcjy_04/20260104/665a9f94e26543cf8a2be770a2d4af61.html)、[`上海市中小学课程标准说明`](https://edu.sh.gov.cn/zcjd_area_257/20200706/0015-xw_18140.html)。边界：这些公开入口用于确认课程 / 教材依据层级，不直接替代教材页内的竖式操作示例。
- [`../../../Specs/a03-vertical-calc/current.md`](../../../Specs/a03-vertical-calc/current.md)：Phase 3 已确认 slot 顺序必须与内置键盘、桌面 `Tab` / `Shift+Tab` 一致；题型层声明 `enabledKeys`、`sanitizeInput` 和 `shouldAutoAdvance`。

默认顺序：

1. 计算前训练格：小数 ÷ 小数先进入扩倍环节，填写扩倍倍数、转换后除数、转换后被除数后点击“确认扩倍”；确认通过后才进入长除法核心板。
2. 长除法核心板：每轮固定按 `商 -> 乘 -> 减 -> 比 -> 落` 推进。
3. 计算后结果格：取近似结果、非循环部分、循环节等后置结构化输入在长除法核心板完成后填写。

核心轮次 slot 顺序：

| 教学步骤 | 学生输入 slot | 自动换格 / 推进规则 |
|---|---|---|
| 商 | `round.{index}.quotientDigit` | 单个数字；填满后进入本轮乘积 |
| 乘 | `round.{index}.product` | 数字串；达到标准长度后进入本轮相减 / 余数 |
| 减 | `round.{index}.remainder` | 数字串；达到标准长度后触发“比”校验 |
| 比 | 无输入 slot | 系统校验 `remainder < divisor`，并可检查 `product <= currentPartialDividend`；失败时停在余数相关位置，不暴露正确值 |
| 落 | `round.{index}.nextPartialDividend` | 非终止轮填写落位 / 补 0 后的新工作数；终止轮没有该 slot |

自动换格原则：

- `delete` 不触发自动换格，继承 Phase 3 键盘规则。
- 单字符商位使用 `sanitizeSingleDigitInput` 或等价策略。
- 乘积、余数、落位 / 补 0 后新工作数使用数字输入；`shouldAutoAdvance` 以标准答案长度为完成条件，避免用粗粒度 `maxLength` 误判。
- 计算前 `trainingFields` / 转换字段按字段顺序自动换格；小数 ÷ 小数扩倍环节的三个转换输入格均允许输入小数点，以支持 `100.0`、`24.0`、`1560.0` 等等值小数，并暴露“转换后除数未化为整数”的错误；扩倍未确认通过前不注册 / 不展示长除法核心板。
- 小数 ÷ 小数的非高档过程提示只用于扩倍确认提交后的即时错误提示，不泄露正确值；高档不显示过程提示，扩倍确认提交有误时直接进入失败反馈，并按训练格明细说明具体错因。
- 计算后 `resultFields` 按展示顺序进入；只有存在下一结构化字段时才自动换格，最后一个结果字段不自动跳。
- 桌面 `Tab` / `Shift+Tab` 必须复用同一份 `orderedInputKeys`，与内置键盘 active slot 顺序完全一致。

Q3 选定方案：题型组件生成 `LongDivisionSlotSpec[]` 或等价内部结构，再映射成 `MathInputSlot[]`；键盘层只消费 slot，不编码长除法语义。

| `long-division-product` | 乘积填写错误 | 商位 × 除数的乘积填错 | 否 |
| `long-division-remainder` | 相减 / 余数错误 | 当前相减结果或余数填错 | 否 |
| `long-division-compare` | 余数大小判断错误 | “比”校验失败，例如余数仍大于或等于除数，或乘积超过当前被除数段 | 否 |
| `long-division-next-partial-dividend` | 落位 / 补 0 错误 | 落下一位或补 0 后形成的新工作数填错 | 否 |

|---|---|---|---|
| `decimal-divisor-scale-factor` | 扩倍倍数错误 | 小数 ÷ 小数的扩倍倍数填错 | 是 |

| `decimal-divisor-scaled-dividend` | 转换后被除数错误 | 被除数同步扩倍后的数填错 | 是 |
| `approximation-result` | 取近似结果错误 | 保留 X 位小数结果填错 | 是 |

展示规则：

- 过程格错误只展示类别列表，不展示“你填 X / 正确是 Y”，避免直接泄露竖式过程答案。
- 扩展训练 / 结构化字段错误展示字段类型、用户值和正确值；空值沿用 Phase 3 口径显示“未填写”。
- 小数 ÷ 小数转换区的扩倍确认例外处理：非高档提交错误时，转换后除数 / 被除数只提示“填写有误”，不在答题过程中展示正确值；高档提交错误时直接判本题过程失败，失败面板展示字段名、用户值和正确值；转换值正确后才进入竖式板。
- 失败反馈主说明按错误类型选择短句：过程格错说明竖式过程有误；扩倍字段错说明扩倍结果有误；计算后结果字段错说明结果表达有误；过程格与结构化字段同时错时说明两类字段都有误。不得使用无法对应具体错误类型的泛化文案，也不得出现行动要求。
- 如果过程格和扩展字段同时错，统一失败面板先展示总体文案，再同时列出过程类别和字段明细。
- 错题本展示同一份结构化错因摘要和字段明细，但不新增“重做完整长除法 UI”“同类题推荐”“回到相关关卡”等行动路径。
- 历史记录 UI 不展示错因；底层 attempt / history / wrongQuestion 继续可选保留 `failureReason` / `failureDetail`，用于未来分析。
- 最终答案错误时按普通错答展示，不额外展示过程格类别或正确过程值。

### 5.2 数据结构草案

Q1 对应的最小结构草案：

```ts
interface LongDivisionRound {
  index: number;
  currentPartialDividend: string;
  sourceDigit?: string;
  sourceKind: 'initial' | 'bring-down' | 'append-zero';
  expectedQuotientDigit: string;
  expectedProduct: string;
  expectedRemainder: string;
  expectedNextPartialDividend?: string;
  isTerminal: boolean;
}
```

说明：

- `currentPartialDividend` 是本轮开始时系统展示的工作数。
- `sourceDigit` / `sourceKind` 用于解释下一轮来源：初始段、落下一位、补 0。
- `expectedNextPartialDividend` 是学生填写“落位 / 补 0 后形成的新工作数”的标准答案；终止轮可为空。
- “比”不需要存成学生输入字段；标准过程生成函数可以保留派生校验信息，例如 `expectedRemainder < coreDivisor`，供 UI 阻止无效落位和 Q5 错因分类使用。
- 实现时命名可调整，但必须保留“来源提示”和“学生填写的新工作数”两个概念，避免把落位 / 补 0 训练误写成系统自动填充。

Q2 对应的任务类型草案：

```ts
type LongDivisionTaskKind =
  | 'core'
  | 'decimal-dividend'
  | 'decimal-divisor-conversion'
  | 'approximation'
  | 'cyclic';

interface LongDivisionTask {
  kind: LongDivisionTaskKind;
  originalExpression: string;
  coreDividend: string;
  coreDivisor: string;
  rounds: LongDivisionRound[];
  trainingFields?: LongDivisionTrainingField[];
  resultFields?: LongDivisionResultField[];
}
```

说明：

- `coreDividend` / `coreDivisor` 是真正进入 `LongDivisionBoard` 的除法对象；小数 ÷ 小数会先让学生在转换区填写扩倍后的除数 / 被除数并提交“确认扩倍”，且只有转换后除数可归一为整数、转换后除数 / 被除数均正确时才派生竖式板结构。
- 小数 ÷ 小数的转换后被除数可为整数或小数；扩倍环节允许三个转换输入格都输入小数点并按数值归一比较；仍是小数时，竖式板据此固定小数点列；若为整数且除尽，商行不显示小数点；两个类型在同一个小数 ÷ 小数预览场景内切换验证。
- 新生成的除法 UI 化题目必须通过 `data.longDivisionBoard` 或等价显式字段进入长除法分支；仅凭 `operation='÷'` 不足以认定为新题型，以免旧存档、旧错题和旧历史回放被误路由。
- `trainingFields` / 转换字段用于扩倍转换等计算前训练格。
- `resultFields` 用于取近似结果、非循环部分、循环节等计算后结构化输入。
- 具体字段枚举中与输入顺序有关的部分已由 Q3 固定；错因枚举与展示边界已由 Q5 固定。

- 若后续认为 `trainingFieldMistakes` 命名不足以覆盖后置结果字段，可在实现时抽更中性的 `structuredFieldMistakes`，但必须兼容当前 `PracticeFailureDetail` 展示 helper 或同步迁移 helper / tests。

### 5.3 不采用方案

Q3 已排除的方案：

| 方案 | 不采用原因 |
|---|---|
| 按视觉从左到右、从上到下自动生成 slot 顺序 | 长除法竖式的视觉位置不等同于计算顺序；会让内置键盘、桌面 `Tab` 和学生实际笔算步骤错位。 |
| 把“比”做成一个学生输入格 | “比”是判断余数是否小于除数的检查动作，不是竖式中的书写结果；做成输入格会增加无意义操作。 |
| 只填商位和最终答案，乘积 / 余数 / 落位系统代填 | 与 Q1 已确认的全过程填写相冲突，无法训练真实竖式除法过程。 |

Q4 已排除的方案：

| 方案 | 不采用原因 |
|---|---|
| `LongDivisionBoard` 返回裸 `boolean` | 与 Phase 1 已确认的新题型板约束冲突；无法承载长除法过程格、训练格和后置结果格错因。 |
| 新增一套只给长除法使用的提交协议 | 会让 `Practice`、错题本、历史记录和同步合并增加并行分支，风险高于收益。 |
| 长除法 `failProcess` 不带 `failureDetail`，只写 `failureReason` | `failureReason` 粒度不足，无法支持 Q5 的错因展示，也可能让旧 fallback 文案误导学生。 |

正式开发前审核已排除的工程方案：

| 方案 | 不采用原因 |
|---|---|
| 在 `Practice.tsx` 直接识别长除法并挂载独立组件 | 会绕开现有 `VerticalCalcBoard` 聚合入口，与多行乘法已采用的内部数据分支不一致，并增加 `handleVerticalComplete` 的上层分支。 |
| 用 dev-only feature flag 控制除法生成器升级 | 当前风险主要来自旧数据回放而不是灰度开关；更低风险的做法是新题用显式 `longDivisionBoard` 数据字段，旧 `numeric-input + operation='÷'` 缺少该字段时保留旧路径，并用回放测试保护。 |

Q5 已排除的方案：

| 方案 | 不采用原因 |
|---|---|
| 过程格错误展示用户值和正确值 | 会直接泄露长除法中间步骤答案，削弱学生复盘过程的训练价值；也与 v0.5 预研和 current spec 冲突。 |
| 训练 / 结构化字段只显示笼统类别 | 扩倍、取近似、循环节这类字段是规则理解错误，展示用户值和正确值更利于定位错误位置；Phase 3 训练格已采用该口径。 |
| 历史记录 UI 展示完整错因 | 历史记录承担结果回顾，不承担错因展开；展示会增加页面密度和移动端溢出风险。 |
| 本轮新增错题本重做长除法流程 | 已超出 `BL-010` 范围；当前目标是保存和展示错因，不新增错题行动路径。 |

### 6.3 容量约束

- 最低移动验收宽度为 375px；实现 QA 至少覆盖 375x812 和 390x844 两个手机竖屏视口，390x844 用于对照 Phase 3 既有键盘基线。
- 非键盘区 / 答题可用区高度必须 ≥60%；内置键盘固定在 viewport 底部，不随题卡滚动。
- 答题内容必须保留底部 padding 或 safe-area 空间，当前 active slot、确认按钮、失败反馈面板不能被固定键盘永久遮挡。
- 长除法核心板必须支持除数 1 / 2 / 3 位；循环小数题池沿用预研上限：循环节最长 3 位、非循环部分为完整前缀且最多 2 位小数、展示字符串总长不超过 8 个字符；超限题不进入循环节识别题池。
- 长除法板最大宽度纳入自适应：空间不足时先降低竖式板字号但不低于 `12px`，再压缩数字间距，再压缩除号 / 除数区，再压缩竖式板和页面边界；仍不足时才允许板内局部横向滚动。当前 UI 审核预览已用 `14 ÷ 135` 极限循环小数样例验证 390px CSS 宽度下不需要左右滑动。
- 长除法板在手机竖屏中优先保证当前轮和当前 active slot 可见；历史轮次保留在同一竖式板内，按 UI 审核稿采用缩放、压缩间距、纵向滚动和必要时局部横向滚动的组合策略。
- 前置训练格按 Q2 已确认的题型转换顺序出现在长除法核心板之前；小数 ÷ 小数的转换区完成后再生成长除法核心板；后置结构化结果格在核心板完成后填写。摆放位置和视觉层级以已确认 UI 审核稿为准。
- 同一训练 / 结构化区内的输入框宽度保持统一，不随标准答案位数变化；输入容量由 `maxLength`、`sanitizeInput` 与等值归一逻辑控制。

### 6.4 响应式与遮挡规则

- 页面不能出现全页横向滚动；长除法板如确需横向承载，必须限制在板内局部滚动区，并保留当前 active slot 可见。
- 输入格、按钮、反馈字段、循环小数展示串和错误明细文本不能溢出父容器；长 token / 表达式需使用换行、缩放或局部滚动策略。
- 反馈面板展示 Q5 已确认的 `processCategories` 与 `trainingFieldMistakes`：过程类别可压缩为短标签 / 多行列表；训练 / 结构化字段明细允许换行展示用户值和正确值，不强制单行。
- 移动端默认使用内置键盘，不依赖系统键盘作为首选输入路径；桌面端必须能不依赖内置键盘，使用真实键盘、`Tab`、`Shift+Tab` 完成输入。
- active slot 需要可见焦点态和可触达区域；当用户滚动页面后，键盘仍贴底，题卡内容可滚动到键盘上方。

### 6.5 验收边界

- Playwright / 截图至少覆盖：375x812、390x844、桌面 1366x768 或 1440x900。
- DOM 度量至少检查：非键盘区高度占比 ≥60%；关键容器 `scrollWidth <= clientWidth`；如长除法板存在局部滚动区，需限定检查该局部容器而非全页；当前 active slot 的 bounding box 不被键盘或反馈面板覆盖。
- 视觉 QA 至少覆盖：普通整数除法、D3 商中间 0、多轮余数传递、小数 ÷ 小数前置训练格、取近似后置结果格、循环小数字段、失败反馈面板。
- 本 Q6 的预览 UI 方案已由用户确认，可作为正式实现参照；生产代码实现完成后仍需用截图 / Playwright / 拟真人工 QA 验收，不能用预览审核结论替代功能 QA。
