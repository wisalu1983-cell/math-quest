## 1. 审核目标

本稿只确认 `BL-010` 的 UI 方案，不提前实现正式长除法判定逻辑。

审核范围：

- 长除法核心板形态：采用专用竖式 UI，商位在长除号上方，除数 / 被除数在竖式主体内，乘积、减线、余数、落位 / 补 0 都在同一块竖式板里完成。
- 数位对齐：题干被除数、商位、乘积、减后工作数全部使用同一套固定数字列；每格只承载一位数字。
- 长除号参考：按标准试卷 / worksheet 的长除号处理，除数在左，左侧采用右括号式弧线分隔，细横线覆盖被除数上方，避免直竖线或厚重圆角“厂字头”。
- 五类题型 UI：整数除法、小数 ÷ 整数、小数 ÷ 小数、取近似、循环小数。
- 输入交互：slot 顺序、自动换格、桌面 Tab、内置键盘。
- 反馈交互：过程格类别反馈、训练 / 结构化字段明细反馈。
- 移动端容量：375px 竖屏下，答题区、反馈面板、固定底部键盘不互相遮挡。

## 2. 推荐方案

采用“专用长除法竖式板 + 外围训练格 / 结果格”的方案。

- 整数除法和转换后的整数除法，核心答题都进入专用 `LongDivisionBoard`，不把商、乘、减、落拆成普通训练格。
- 商位输入在长除号上方；乘积、相减结果、落位 / 补 0 后新工作数在竖式主体里输入。
- 竖式主体内只保留数字格、减号、横线和小数点；“第几轮”“商乘减比落”等教学提示放到旁侧状态或无障碍说明，不压进笔算表内。
- 竖式板不一次性展示全部待填格；只显示当前已到达 / 已填写的字段，后续乘积行、减线、余数与落位行、下一轮商位按输入顺序逐步向下长出。
- 竖式板所有数字格统一略窄，小数点作为独立符号放在相邻数字格间隙中，保证左右都有清晰留白。
- 数字列间隙分两套：普通相邻数字列使用常规间隙；存在小数点时，只有小数点左右两个数字格之间使用小数点专用间隙，并把小数点放进该间隙轨道，保证桌面和移动端缩放后仍与数字格中心线对齐。
- 长除号弧线使用独立括号列；竖式板上的除数 / 被除数由转换区输入派生，被除数首位从括号列右侧开始，不能与弧线重叠。
- 计算前训练格和竖式板都按主答题区中心线居中，避免板面偏左、右侧空白过大。
- 非终止轮不再出现“余数格 + 新工作数格”的重复填写；减线下方同一行直接填写余数与落位后的工作数，例如 `9 - 8` 后直接在下一行填 `13`，再进入下一轮。
- 小数 ÷ 整数中，商的小数点由系统预置，必须与被除数小数点垂直对齐，并作为独立符号放在两个答案数字格之间，不占用数字格。
- 整数 ÷ 整数中，竖式板默认不显示商小数点；只有当被除数个位处理完仍有余数且本题要求继续算小数时，系统才在商的个位后自动出现小数点。
- 小数 ÷ 小数拆成两步：先在扩倍环节填写“除数扩大几倍、转换后除数、转换后被除数”，点击“确认扩倍”提交；通过后进入实际计算环节，再生成长除法竖式板。
- 扩倍环节三个输入格的小数点键始终可用；等值小数可按数值归一后通过，例如 `100.0` 等同 `100`、`24.0` 等同 `24`。
- 转换后除数必须可归一为整数；如果用户没有把除数处理成整数，扩倍环节直接提示“转换后除数必须是整数”，不进入竖式板。
- 非高档下扩倍提交有误时，只提示对应字段“填写有误”，不直接给出正确值，并停留在扩倍环节等待再次确认；高档下扩倍提交有误时，本题直接判为过程失败，并在失败反馈中说明具体错因。
- 扩倍确认通过后，再按转换后被除数走“小数被除数预置小数点”或“整数被除数必要时自动出现小数点”两种模式。
- 实际计算环节提交答案时走通用失败反馈：长除法过程格、取近似 / 循环结构等触发错因统一展示。
- 扩倍后的被除数可以是整数，也可以仍是小数；转换区允许被除数填写小数点，竖式板在转换值正确后按被除数固定小数点位置并生成数字列。
- 循环小数后置结果格中，“非循环部分”填写完整前缀，包含整数部分和小数点，例如 `0.1`；“循环节”填写最短重复片段，例如 `037`。
- 循环小数的标准格式答数按中国小学考卷 / 练习册常见写法展示循环点：循环节 1 位时在该位上方打点，循环节多位时在首位和末位上方打点；预览面板不展示括号格式。
- 历史轮次留在同一块竖式板内；当前 active slot 所在轮次高亮。
- 前置训练格只用于小数 ÷ 小数扩倍；后置结构化结果格只用于取近似、循环小数等核心计算后的结果表达。
- 内置键盘复用 Phase 3 `MathInputSlot` 协议，题型组件只声明 slot。

选择原因：

- 375px / 390px 手机竖屏仍以专用竖式板为主；空间不足时按通用自适应顺序处理：先降低竖式板字号但不低于 `12px`，再压缩数字间距，再压缩除号 / 除数区，再压缩竖式板内边距和页面边界；仍不足时才允许竖式板局部横向滚动。
- 符合 Phase 4 已确认的全过程填写：学生仍要填商位、乘积、余数、落位 / 补 0。
- “比”保留为系统校验节点，不增加无意义输入。
- 与 Phase 3 已实现的固定底部键盘、自动换格、结构化错因能直接衔接。

## 3. 预览场景

| 场景 | 样例 | 审核重点 |
|---|---|---|
| 整数多轮 | `936 ÷ 4` | 多次余数传递；完整“商、乘、减、比、落”顺序 |
| 商中间 0 | `824 ÷ 4` | 不够除时商 0；落下一位后形成新工作数 |
| 小数 ÷ 整数 | `5.76 ÷ 3` | 商的小数点系统预置，不作为输入格 |
| 小数 ÷ 小数 | `15.6 ÷ 0.24` / `1.234 ÷ 0.04` | 同一个预览场景内切换“转换后整数 / 转换后仍有小数”；先确认扩倍再进入竖式板；扩倍格小数点始终可用；非高档转换值填错只提示字段错误、不泄露正确值、不进入竖式板；高档转换值填错直接进入失败反馈 |
| 取近似 | `8.5 ÷ 3` 保留两位小数 | 算到保留位后一位，最后只填近似结果 |
| 循环小数 | `14 ÷ 135` | 严格按展示串 8 字符上限模拟极限：非循环部分 `0.1`、3 位循环节 `037`；竖式板中循环节至少出现 2 次后，核心计算后填写完整非循环部分和循环节，并由系统显示中国循环点格式标准答数 |

## 4. 交互设计

### 4.1 输入顺序

统一顺序：

```text
小数 ÷ 小数扩倍环节：除数扩大倍数、转换后除数、转换后被除数（仅该类题有）
→ 点击“确认扩倍”
→ 扩倍通过后生成实际计算用竖式板
→ 长除法竖式板第 1 轮 商位
→ 第 1 轮 乘积
→ 第 1 轮 余数与落位后的工作数（同一行）
→ 系统校验“比”
→ 下一轮 ...
→ 取近似 / 循环小数后置结构化结果格（仅该类题有）
```

规则：

- 单位商位达到 1 位后自动进入乘积。
- 乘积、余数与落位同一行达到标准答案长度后自动进入下一 slot。
- 竖式板每次只露出当前步骤需要的格子；已完成的历史步骤保留在板内，未到达的后续轮次不提前显示。
- `delete` 不自动换格。
- 桌面 `Tab` / `Shift+Tab` 使用同一份 slot 顺序。
- 移动端默认固定底部内置键盘；不依赖系统键盘作为首选输入方式。
- 题底操作区保留统一规格的“提交”和“重置”按钮；重置清空当前题所有输入、关闭反馈，并回到第一格未填写状态。
- 进入失败反馈态后收起内置键盘，反馈面板自动滚入可见区域。

### 4.2 题型扩展格

| 题型 | 前置 / 后置格 | 展示方式 |
|---|---|---|
| 小数 ÷ 整数 | 无额外输入；商小数点系统预置 | 核心板上方给出绿色提示 |
| 小数 ÷ 小数 | 扩倍倍数；转换后的除数 / 被除数 | 同一转换区连续填写并点击“确认扩倍”；三个输入格小数点键始终可用；同一预览内切换转换后被除数为整数 / 仍有小数两类；转换后除数仍为小数时先警告并阻止进入竖式板；非高档错填转换值时只提示字段填写有误，不给正确值，也不进入竖式板；高档错填转换值时直接判本题失败并展示具体错因 |
| 取近似 | 保留位数结果 | 核心板后的结果格区 |
| 循环小数 | 非循环部分、循环节、标准格式答数 | 先在核心板完成整数位 `0`、乘积 `0`、补 0 后工作数；循环节至少出现 2 次后，结果格填写完整非循环前缀（如 `0.1`）和循环节（如 `037`），系统按中国循环点格式显示标准答数 |

扩展格视觉规则：

- 同一训练 / 结构化区内的输入框使用统一宽度，不随标准答案位数变宽变窄；位数差异由 `maxLength` / sanitize 控制，视觉上保持整齐。

### 4.3 失败反馈

过程格错误：

- 展示类别，不展示正确中间值。
- 例：`商位判断错误`、`乘积填写错误`、`落位后新工作数错误`。

训练 / 结构化字段错误：

- 展示字段名、用户值、正确值。
- 例：`扩倍倍数错误：你填 10，正确是 100`。
- 小数 ÷ 小数扩倍环节的非高档错误提示不泄露正确值；高档不做扩倍错误重试循环，点击“确认扩倍”后若有误直接进入失败反馈，并按训练格明细展示用户值和正确值。

反馈面板主说明按错误集合生成短句，不使用“最终答案可能已经接近”这类泛化文案，也不出现行动要求：

- 只有竖式过程格错误：`本题未通过：竖式过程有误。`
- 只有扩倍转换字段错误：`本题未通过：扩倍结果有误。`
- 只有计算后结果字段错误：`本题未通过：结果表达有误。`
- 过程格和结构化字段同时错误：`本题未通过：竖式过程和结构化字段都有误。`

错题本：

- 复用同一份结构化错因摘要。

历史记录 UI：

- 不展示错因，只保留底层数据。

## 5. 移动端容量策略

- 页面不能出现全页横向滚动；专用竖式板优先整体缩放，达到最小 scale 后仍无法容纳时才允许板内受控横向滚动。
- 当前 active slot 所在轮次高亮，必要时实现自动滚动到可见区域。
- 固定底部键盘保留约 `14rem` 底部空间，避免提交按钮被遮挡；反馈态隐藏键盘，避免反馈面板与键盘抢占空间。
- 手机竖屏中，竖式板按可用宽度和键盘上方剩余高度共同计算 scale；桌面优先展示更接近纸面笔算的完整板面。
- 竖式板最大宽度纳入自适应：空间不足时先调字号、再调间距、再调除号区和板 / 页面边界；当前极限循环小数样例在 390px CSS 宽度下不需要竖式板左右滑动。
- 循环小数结果格展示长度沿子计划限制：循环节最长 3 位、非循环部分为完整前缀且最多 2 位小数、总展示字符串不超过 8 字符；标准格式答数用循环点，不用括号。
- 循环小数竖式板的结束条件改为“循环节至少出现 2 次”。这与结果格的“循环节最长 3 位”不冲突；但它比“首次重复余数即结束”多展示一轮循环，需要在移动端容量和题池过滤中单独校验。

## 6. 已记录问题

- `BUG-PREVIEW-001`：桌面物理键盘输入后，active slot 视觉状态会自动进入下一格，但浏览器真实焦点仍留在旧 input，导致继续打字写不进下一格。预览稿已修正为桌面端 `activeSlotId` 变化时同步 focus；正式 `BL-010` 实现必须继承这条焦点规则。
- `BUG-PREVIEW-002`：转换后被除数为整数且除尽时，商行曾提前显示小数点；预览稿已改为小数 ÷ 整数系统预置小数点、整数 ÷ 整数只在个位后仍需继续小数时自动出现小数点，小数 ÷ 小数先校验转换后除数必须为整数。
- `BUG-PREVIEW-003`：转换后除数填错曾直接提示“应为 X”，转换后被除数填错曾无提示且不进入竖式板；预览稿已改为非高档只提示字段填写有误、不泄露正确值，并在扩倍环节停住；高档提交错误直接进入失败反馈。
- `BUG-PREVIEW-004`：扩倍环节曾把是否允许输入小数点绑定到标准答案是否带小数，导致 `100.0` / `24.0` / `1560.0` 这类等值小数无法作为扩倍答案通过；预览稿已改为三个扩倍输入格小数点始终可用，并按数值归一比较。
- `BUG-PREVIEW-005`：390px CSS 宽度下极限循环小数样例曾需要竖式板局部左右滑动；预览稿已加入通用自适应压缩档位，极限样例在 390px 下不需要左右滑动。
- `BUG-PREVIEW-006`：循环小数结果格曾把“非循环部分”理解为小数点后的非循环数字 `1`，且标准答数用括号格式 `0.1(037)`；预览稿已改为填写完整非循环前缀 `0.1`，并按中国常见循环点格式显示标准答数。

## 7. 本轮已确认与仍需确认

已确认：

1. 小数 ÷ 小数采用两步：扩倍环节先提交，确认正确后进入实际计算环节。
2. 扩倍环节三个输入格的小数点键始终可用；是否带小数不由标准答案格式决定。
3. 非高档扩倍错填只提示字段错误并停留在扩倍环节；高档扩倍错填直接判本题过程失败并展示具体错因。
4. 循环小数“非循环部分”填写完整前缀，标准格式答数采用中国循环点格式，不采用括号格式。
5. 极限循环小数样例使用通用响应式压缩策略，390px CSS 宽度下不需要竖式板左右滑动。
6. 2026-04-30 用户确认当前预览版暂无其他问题，可作为正式实现参照。

仍需后续关注：

1. 正式生产实现时，需要把预览中的动态轮次生成、响应式压缩和循环点显示抽到可测试的正式组件 / helper 中。
2. 功能实现完成后，仍需按 Phase 4 QA 覆盖 375x812、390x844 和桌面视口；当前 UI 审核通过不替代生产功能 QA。
