## 一、背景

`ISSUE-067` 的现象是：多行乘法竖式中，学生最终积填对，但部分积、合计过程格或小数乘法训练格填错时，系统会判错；统一失败面板只显示正确答案和解析，不说明“为什么答案对还没通过”。这会让用户误以为判题 bug。

v0.5 预研已确认：判定规则不改，过程 / 训练格错误仍不通过；要修的是结构化错因传递和展示。Phase 3 要把这个链路做成可复用能力，Phase 4 长除法严格过程判定才能直接沿用。

### 工作脉络

1. v0.4 legacy 单行竖式已具备 `failureReason='vertical-process'`，但只覆盖进位 / 退位格。
2. v0.4 多行乘法竖式落地时，`MultiplicationVerticalBoard` 只返回 `boolean`，没有把部分积、合计行或小数训练格错误传给 Practice。
3. v0.5 Phase 1 已确认后续题型板不得继续使用裸 `boolean` 作为提交结果，必须返回带错因的 payload。
4. Phase 3 先补多行乘法结构化错因；Phase 4 长除法再复用同一套失败明细、反馈面板和错题本展示能力。

## 二、问题定义

### 现状

- `src/components/MultiplicationVerticalBoard.tsx` 的 `onComplete` 只返回 `boolean`。
- `src/pages/Practice.tsx` 收到 `false` 后提交 `"竖式计算有误"`，没有传入 `failureReason` 或明细。
- 统一失败面板目前只识别 legacy 单行竖式的 `lastFailureReason === 'vertical-process'`。
- `WrongBook.tsx` 也只展示旧的“进位/退位格填写错误”。

### 用户影响

- 学生看到“正确答案是 4730”，但自己最终积也是 4730，会认为系统错判。
- 家长或测试者无法从错题本知道问题出在过程格还是小数训练格。
- Phase 4 长除法若继续使用严格过程判定，缺少错因会放大挫败感。

## 三、产品与交互口径

### 3.1 判定不变

- 最终答案错误：未通过，按普通错答展示正确答案和解析。
- 最终答案正确，但多行乘法过程格错误：未通过，展示过程类别文案。
- 最终答案正确，但小数乘法训练格错误：未通过，展示具体训练格错误。
- 最终答案正确，过程格和训练格都正确：通过。

### 3.2 展示优先级

| 场景 | 统一失败面板展示 |
|---|---|
| 最终答案错误 | 现有正确答案 + 解析；不额外暴露过程格正确值。 |
| 最终答案正确 + 只有过程格错 | “未通过原因：竖式过程填写错误。虽然最终答案正确，但本题要求竖式过程也正确。” |
| 最终答案正确 + 只有训练格错 | 展示每个训练格的类型、用户值、正确值。 |
| 最终答案正确 + 过程格和训练格都错 | 先展示过程类别，再展示训练格明细。 |

### 3.3 过程格分类

首版不逐格暴露过程格的用户值 / 正确值，只显示类别：

| code | 文案 |
|---|---|
| `multiplication-partial-product` | 部分积填写错误 |
| `multiplication-total` | 竖式求和过程填写错误 |
| `multiplication-process` | 竖式过程填写错误 |

若实现时难以低风险区分 partial 与 total，可以首版统一返回 `multiplication-process`，但必须保留扩展空间。

### 3.4 训练格分类

训练格展示具体对比：

| code | 来源字段 | 文案 |
|---|---|---|
| `decimal-operand-a-places` | `operand-a-decimal-places` | 被乘数小数位数错误 |
| `decimal-operand-b-places` | `operand-b-decimal-places` | 乘数小数位数错误 |
| `decimal-move` | `decimal-move` | 小数点移动位数错误 |

展示格式：

```text
小数点移动位数错误：你填 1，正确是 2
```

用户值为空时显示“未填写”；不显示 `undefined`、空字符串或技术 key。

### 3.5 错题本与历史记录

- 当前反馈面板：显示本次具体错因。
- 错题本：显示错因摘要和训练格明细。
- 历史记录 UI：不展示错因。
- 底层 `QuestionAttempt` / `WrongQuestion` 可以保留可选明细字段，用于未来分析。

## 四、工程实现方案

### 4.1 类型方案

保留旧字段，新增可选结构化明细：

```ts
export type PracticeFailureReason =
  | 'wrong-answer'
  | 'vertical-process'
  | 'vertical-multiplication-process'
  | 'vertical-training-field';

export interface PracticeFailureDetail {
  reason: PracticeFailureReason;
  source: 'legacy-vertical' | 'vertical-multiplication' | 'long-division';
  message: string;
  processCategories?: Array<{
    code: string;
    label: string;
  }>;
  trainingFieldMistakes?: TrainingFieldMistake[];
}
```

实现时可按代码风格调整命名，但必须满足：

- `failureReason` 仍是简单字符串，兼容旧 wrongQuestions / history。
- `failureDetail` 为可选对象，不触发存档版本 bump。
- 没有 `failureDetail` 的旧数据仍能按旧文案展示。

### 4.2 提交 payload

`MultiplicationVerticalBoard` 不再只返回 `boolean`，而是返回结构化 payload。建议抽出通用 payload，让后续长除法复用：

```ts
export type PracticeBoardCompletePayload =
  | { result: 'pass'; answer: string }
  | { result: 'failWrongAnswer'; answer: string; failureReason: 'wrong-answer' }
  | {
      result: 'failProcess';
      answer: string;
      failureReason: PracticeFailureReason;
      failureDetail: PracticeFailureDetail;
    };
```

legacy `VerticalCalcBoard` 可以继续映射为旧 `vertical-process`，但 `Practice` 的处理逻辑应统一消费 payload。

### 4.3 错误分析落点

| 文件 | 改动 |
|---|---|
| `src/components/MultiplicationVerticalBoard.tsx` | 根据 `orderedInputKeys`、`expectedByKey`、`finalAnswerKey` 分析最终答案、过程格、训练格错误，返回 payload。 |
| `src/pages/Practice.tsx` | `handleVerticalComplete` 消费 `failureDetail`；失败面板使用统一渲染 helper。 |
| `src/store/index.ts` | `SubmitAnswerOptions`、`QuestionAttempt`、`WrongQuestion` 支持可选 `failureDetail`；`lastFailureDetail` 支持当前反馈。 |
| `src/pages/WrongBook.tsx` | 复用错因渲染 helper；旧 `vertical-process` 数据仍显示旧文案。 |
| `src/sync/merge.test.ts` | 增加 merge 保留 `failureDetail` 的测试。 |

### 4.4 渲染 helper

建议新增纯 helper：

- `src/utils/practiceFailureDisplay.ts`
- `src/utils/practiceFailureDisplay.test.ts`

职责：

- 输入 `failureReason` + 可选 `failureDetail`。
- 输出标题、说明、过程类别列表、训练格明细。
- 处理旧数据 fallback。
- 统一 Practice 反馈面板与 WrongBook 文案。

### 4.5 不采用的方案

| 方案 | 放弃原因 |
|---|---|
| 只把 `false` 改成 `failureReason='vertical-process'` | 能显示旧文案，但无法区分多行乘法、训练格，也不能服务长除法。 |
| 过程格逐格显示用户值和正确值 | 与预研口径冲突，会把过程答案直接暴露给学生，降低训练价值。 |
| 新增必填字段并 bump 存档版本 | 本问题不需要破坏兼容；可选字段足够。 |
| 只在当前反馈面板显示，不进错题本 | 与预研口径冲突；错题本需要作为未来分析数据来源。 |

### 4.6 实现架构决策门

| 维度 | 结论 |
|---|---|
| 模块 ownership 与依赖方向 | `MultiplicationVerticalBoard` owns 本板错误分析；`Practice` owns 统一反馈面板；`store/index.ts` owns session attempt、pending wrongQuestions 和当前反馈状态；`WrongBook` owns历史错题展示。 |
| 类型与 API 契约 | 保留旧 `failureReason` 字符串，新增可选 `failureDetail`；`MultiplicationVerticalBoard` 返回结构化 payload，不再只返回 `boolean`。 |
| 持久化 / 存档 / 版本迁移 | `failureDetail` 只作为可选字段进入 attempt / wrongQuestion；旧数据可读，无存档版本 bump；同步合并保留该字段。 |
| 标识符命名空间与向后兼容 | 新 reason 使用 `vertical-*` 命名空间；旧 `wrong-answer`、`vertical-process` 保留；训练格 code 不暴露组件内部 key 给用户。 |
| UI 容量与响应式约束 | 失败面板和错题本在 375px 竖屏下不能溢出；过程类别和训练格明细分块展示，避免一段长文挤压继续按钮。 |
| 测试与 QA 映射 | display helper 单测、store 单测、sync merge 测试、整数乘法和小数乘法浏览器验收共同覆盖。 |
