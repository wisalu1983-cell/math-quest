## 三、开工对齐结论

### 1. 文档拆分

Phase 1 只保留本文件作为开工对齐入口。后续详细设计按 Phase 落到独立子计划：

| Phase | 子计划主题 | 文档职责 |
|---|---|---|
| Phase 2 | `BL-009` 竖式题样本质量诊断与过滤规则 | 先跑样本诊断，再定过滤规则和落点。 |
| Phase 3 | `BL-011` 内置键盘 + `ISSUE-067` 错因反馈基础设施 | 建立统一计算输入模型、系统键盘兜底、结构化错因链路。 |
| Phase 4 | `BL-010` 竖式除法 UI 化答题 | 在 Phase 3 基础上实现动态长除法板和扩展训练格。 |
| Phase 5 | Release Gate 与 Living Spec 回写 | 汇总 QA、发布门禁、current spec 回写和 Backlog / Issue 收口。 |

### 2. 技术 ownership 初判

| 能力 | ownership | 依赖方向 |
|---|---|---|
| 竖式题样本过滤 | `src/engine/generators/vertical-calc.ts` 为生成器主归属；`src/constants/campaign.ts` 只承接 lane 过滤或关卡约束 | 先诊断生成器输出，再决定是否进入 generator helper 或 campaign lane。 |
| 内置计算键盘 | Practice 输入基础设施归属，优先落在 `src/pages/practice-input-state.ts` 旁的新 hook / component；题型组件只声明输入格与动作 | 键盘写状态，不让各题型散落处理系统键盘策略。 |
| 错因反馈 | 类型归属 `src/types/index.ts`；会话归属 `src/store/index.ts`；展示归属 `src/pages/Practice.tsx` 与 `src/pages/WrongBook.tsx` | 题型板返回结构化结果，`Practice` 统一进入反馈和错题链路。 |
| 多行乘法竖式 | `src/components/MultiplicationVerticalBoard.tsx` 保持板内判定，但回调从 boolean 升级为结构化 payload | 不改变判定规则，只补齐失败原因。 |
| 长除法 UI | 新组件 / 新 engine helper 独立承载，不继续膨胀 legacy `VerticalCalcBoard` | 复用 Phase 3 的输入与错因能力；不依赖乘法板内部实现。 |

### 3. 类型与 API 约束

- 后续新题型板不再用单纯 `boolean` 表示提交结果；必须返回带结果类型、答案、错因的 payload。
- 错因需要区分两类：
  - 过程格错误：只显示类别，不显示用户值和正确值。
  - 训练格错误：显示具体类型、用户值和正确值。
- `PracticeFailureReason` 需要扩展命名空间，但旧值 `wrong-answer`、`vertical-process` 保留兼容。
- 如果 Phase 3 需要持久化详细训练格错因，字段必须设计成可选字段；是否需要存档版本升级由实现前的类型方案决定。

### 4. UI 与输入约束

- Chrome Android / Safari iOS 默认内置键盘，不自动弹系统键盘。
- 系统键盘入口必须独立于内置键盘面板存在，避免内置键盘异常时无法兜底。
- 计算输入入口包括普通答案、商余数、多空、表达式 / 等式、竖式格、多行乘法、小数训练格、未来长除法格；选择题 / 判断题不接入。
- 长除法、失败面板、内置键盘必须分别写手机竖屏验收点，不能只以桌面宽度为准。

### 5. QA 映射

| 决策 | 验收方式 |
|---|---|
| 样本过滤先诊断后落规则 | 统计脚本 / 抽样报告 + 自动化样本验证。 |
| 内置键盘默认策略 | Playwright 或拟真浏览器验证 Chrome / Safari 目标路径；其他浏览器允许降级。 |
| 错因结构化 | Store 单测、Practice 反馈测试、WrongBook 展示验证。 |
| 长除法严格过程判定 | engine 单测、组件交互测试、移动端截图或拟真 QA。 |
| Current spec 回写 | Phase 验收后回写，不在开发期提前改写。 |
