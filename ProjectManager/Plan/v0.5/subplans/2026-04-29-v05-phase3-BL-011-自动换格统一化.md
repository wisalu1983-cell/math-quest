## 一、背景

Phase 3 `BL-011` 已建立内置计算键盘、输入槽位协议和槽位级 `enabledKeys` / `sanitizeInput`。但本轮真实体验暴露出两个后续问题：

1. 内置键盘按键当前只写入 active slot；是否自动移动到下一格由各题型组件自行处理。
2. 多行乘法竖式的部分积 / 总积格当前按视觉数组从左到右自动移动，不符合笔算从低位到高位的处理顺序。

本文件是 `BL-011` 的 follow-up 优化文档，目标是把“写入后是否自动换下一格”收敛为通用键盘能力，同时保留题型层对真实书写顺序和完成条件的声明权。

### 工作脉络

1. `PracticeMathKeyboard` 当前在按键点击时执行 `targetSlot.setValue(applyMathKeyboardKey(...))`，键盘层本身不决定下一 slot。
2. `VerticalCalcBoard` 已在本轮临时修复中让内置键盘路径调用 `focusAfterInput()`，修复 legacy 竖式格不自动换格。
3. `MultiplicationVerticalBoard` 已在本轮临时修复中新增 `activateNextSlot()`，修复多行乘法内置键盘不自动换格。
4. 全题型扫描确认，真正不符合现实顺序的是多行乘法竖式的部分积 / 总积行；普通题、商余数、多空、表达式 / 等式、训练格本身没有同类“右到左”顺序问题。
5. 因为自动换格是跨题型输入基础能力，后续不应继续在每个组件内复制 `activateNextSlot()` 这类局部逻辑。

## 二、问题定义

### 2.1 用户可见问题

- 用内置键盘填写多输入格时，部分题型不会自动跳到下一格，或跳格顺序不符合实际书写习惯。
- 多行乘法竖式中，部分积和总积按左到右移动；实际笔算一般从右侧低位开始填写，再向左处理高位。

### 2.2 技术根因

| 层级 | 当前状态 | 问题 |
|---|---|---|
| 通用键盘层 | `PracticeMathKeyboard` 只把按键应用到 active slot 并调用 `setValue` | 没有统一自动换格协议 |
| slot 类型层 | `MathInputSlot` 有 `maxLength`、`enabledKeys`、`sanitizeInput`、`setValue` | 没有“何时视为完成并移动”的声明 |
| 题型组件层 | legacy 竖式和多行乘法各自实现局部换格 | 容易重复、遗漏、顺序不一致 |
| 乘法竖式顺序 | `calculationRows.flatMap(row.cells...)` 按视觉左到右生成 key | 与笔算低位到高位顺序不一致 |

### 2.3 用户影响

- 移动端学生需要手动点下一格，练习节奏被打断。
- 多行乘法按左到右输入时，学生需要反着真实笔算顺序操作，容易产生认知摩擦。
- 若后续长除法继续复制局部逻辑，会把同类问题带进 Phase 4。

## 三、优化目标

- 通用键盘层负责“按键写入后是否自动移动 active slot”的统一执行。
- 题型层只声明三件事：slot 顺序、slot 是否自动换格、slot 完成条件。
- 多行乘法竖式的部分积 / 总积行内顺序改为右到左；行序仍按从上到下。
- 已有 legacy 竖式和多行乘法临时修复要被统一机制吸收，避免长期保留两套内置键盘换格逻辑。
- 表达式、等式、最终答数等多字符输入不自动换格。
- 桌面实体键盘路径保持可用；移动端内置键盘路径和桌面内置键盘辅助路径行为一致。

## 四、可选方案对比

| 方案 | 描述 | 优点 | 风险 / 放弃原因 |
|---|---|---|---|
| A. 每个题型继续自行实现换格 | 在 `VerticalCalcBoard`、`MultiplicationVerticalBoard`、`Practice.tsx` 各自判断下一格 | 改动小，局部见效快 | 会继续复制逻辑；商余数、多空、训练格、长除法容易再次遗漏；不符合 `BL-011` 输入基础设施目标 |
| B. 通用键盘无条件按 `maxLength` 自动换格 | 键盘层发现 `next.length >= maxLength` 就跳下一 slot | 实现简单 | 最终答数、表达式、等式、多位答案会误跳；训练格答案长度不一定等于 `maxLength` |
| C. 通用键盘执行，题型 slot 声明完成条件 | `MathInputSlot` 增加可选 `shouldAutoAdvance` 或等价字段；键盘层统一调用 | 低风险、可扩展；能区分单字符格、多位字段、自由文本；便于长除法复用 | 需要迁移现有局部逻辑，并补较完整回归测试 |

**选定方案：C。**

## 五、选定方案

### 5.1 类型契约

在 `MathInputSlot` 增加可选自动换格声明：

```ts
export interface MathInputSlot {
  id: string;
  label: string;
  value: string;
  maxLength?: number;
  enabledKeys: MathKeyboardKey[];
  sanitizeInput?: (raw: string, previous: string) => string;
  setValue: (next: string) => void;
  shouldAutoAdvance?: (params: {
    key: MathKeyboardKey;
    previousValue: string;
    nextValue: string;
  }) => boolean;
  focusSystemInput?: () => void;
}
```

实现时允许命名微调，但必须满足：

- 默认不自动换格，避免误伤自由文本。
- `delete` 不触发自动换格。
- 完成条件由 slot 声明，键盘层只执行。
- 完成条件依赖的外部信息，例如商的标准长度、`solution.blanks[index]`、`field.answer.length`、`expectedByKey[key]`，由题型组件在构建 slot 时通过闭包捕获进 `shouldAutoAdvance`；键盘层不新增题目答案、字段答案或题型数据参数。
- 键盘层根据当前 `slots` 顺序找到下一 slot，并调用 `onActiveSlotChange(nextSlot.id)`。
- 不在键盘层编码题型语义，例如“不知道部分积是什么”“不知道商余数是什么”。

### 5.2 题型层职责

| 题型 / 输入形态 | slot 顺序 | 自动换格规则 |
|---|---|---|
| 普通单答案 | 单 slot | 不自动换格 |
| 商余数 | 商 → 余数 | `Practice` 从当前题 `solution.answer` 的商部分推导标准长度；商达到该长度时可自动到余数，余数不自动跳 |
| multi-blank | 题面空位顺序 | `Practice` 用 `solution.blanks[index]` 转字符串后的长度作为完成条件；每空达到对应答案长度时自动到下一空 |
| 训练格 | `trainingFields` 顺序 | `Practice` / `DecimalTrainingGrid` 用 `field.answer.length` 作为完成条件；每格达到对应长度时自动到下一格 |
| 表达式 / 等式 | 单 slot | 不自动换格 |
| legacy 竖式 | 沿 `getNextFocus()` 的计算步骤 | 单字符完整后自动换格，仍服从既有三档策略 |
| 多行乘法操作数空格 | 每行左到右 | 单字符完整后自动换格 |
| 多行乘法部分积 / 总积 | 每行右到左，行序上到下 | 单字符完整后自动换格 |
| 小数位数 / 移动位数 | 教学步骤顺序 | 达到预期长度后自动换格 |
| 最终答数 | 单 slot | 不自动换格 |

### 5.3 乘法竖式旧调整是否覆盖

需要覆盖，但不是删除所有乘法组件逻辑。

应覆盖的部分：

- `MultiplicationVerticalBoard` 里专门为内置键盘新增的 `activateNextSlot()` 不应作为长期方案保留。
- 内置键盘 slot 的 `setValue` 不应直接在组件内判断 `if (next) activateNextSlot(key)`。
- 这部分应迁移为 slot 的 `shouldAutoAdvance` 声明，由 `PracticeMathKeyboard` 统一执行下一 slot 激活。

仍保留在乘法组件的部分：

- `orderedInputKeys` 的生成权仍在 `MultiplicationVerticalBoard`，因为只有乘法板知道哪些 row 是操作数、部分积、总积、小数定位字段。
- 部分积 / 总积的行内顺序应在乘法组件里生成成右到左。
- 桌面真实 input 的焦点管理可以继续通过组件内 DOM refs 执行，但应尽量复用同一份 slot 顺序，避免桌面和移动端顺序分叉。

结论：**之前单独为乘法竖式做的“自动换格局部 patch”应被统一自动换格机制替代；乘法竖式仍负责声明正确输入顺序。**

## 八、风险与回滚思路

| 风险 | 说明 | 控制方式 |
|---|---|---|
| 多位答案误跳 | 商、multi-blank、trainingFields 若用 `maxLength` 判断会太粗 | 使用 `expected length` 或题型显式 `shouldAutoAdvance`，默认不跳 |
| 编辑已有值时误跳 | 学生修改中间值时可能再次触发跳格 | `delete` 不跳；只有输入有效非删除键且满足完成条件时跳 |
| 编辑回填再次跳走 | 学生填满格 A 自动跳到格 B 后，点回格 A 修改；再次填满格 A 时，系统仍会按完成条件跳到格 B。这可能符合“填满即继续”的心智，也可能干扰精修答案 | 初版按“再次满足完成条件仍自动换格”实现；拟真 QA 观察是否造成修改答案时的明显停顿或误操作 |
| 桌面和移动端顺序分叉 | 桌面 refs 与内置键盘 active slot 使用不同下一格逻辑 | 同一 `orderedInputKeys` 同时服务 slot 顺序和桌面 focus |
| 乘法预览遗漏 | preview 复制了旧顺序逻辑 | T6 同步预览，或抽共享顺序 helper |
| 过度抽象 | 为少数题型引入复杂状态机 | 只新增 slot 级 `shouldAutoAdvance`，不引入全局焦点编排器 |

回滚思路：

- 若通用机制引发大面积输入回归，可先关闭普通 `Practice.tsx` 多槽位的 `shouldAutoAdvance` 声明，保留 legacy 竖式 / 多行乘法必需路径。
- 若乘法右到左顺序引发意外，可只回滚计算行顺序生成，不影响键盘通用机制。
