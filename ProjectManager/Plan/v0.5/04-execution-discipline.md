## 执行原则

- 先对齐、再实现：Phase 1 只先解决跨 Phase 不冲突的问题；各子项详细设计与实施拆解下沉到 Phase 2 / 3 / 4 子计划。
- 证据先行：涉及题目过滤、浏览器键盘、现有判定规则时，必须先读代码 / 跑诊断 / 查官方资料，不能凭经验写死。
- 低风险优先：能用现有 Practice / A03 架构承接的，不引入独立大框架。
- 严格区分过程格和训练格：过程格错因只显示类别；训练格错因显示具体类型、用户值和正确值。
- current spec 不提前回写：开发期只标 `Spec impact`；phase 验收确认后再回写。

## Phase 1 开工对齐与后续子计划启动门

Phase 1 的文档目标是“别打架规则”，不一次性写完全部详细设计。它必须确认跨 Phase 边界、模块归属、共享数据口径和 QA 映射；具体过滤阈值、键盘 UI 细节、长除法格子模型分别在 Phase 2 / 3 / 4 子计划中细化。

进入任何实现前，Phase 1 开工对齐文档或对应 Phase 子计划必须显式回答：

- 模块 ownership 与依赖方向：题目过滤、输入键盘、长除法核心板、错因反馈分别归属哪个模块。
- 类型与 API 契约：长除法步骤、训练格、错因枚举、键盘按键、过滤规则需要哪些类型。
- 持久化 / 存档 / 迁移：错因数据、错题本记录是否需要新增存储字段或同步策略。
- 标识符命名空间与兼容：新增题型 subtype、错误原因 id、输入组件 id 或 dev 入口是否与现有 id 冲突。
- UI 容量与响应式约束：移动端竖屏、长循环节、内置键盘高度、答题区高度占比、失败面板文案均要有验收点。
- 测试与 QA 映射：每个架构决策至少映射到单测、集成测试、构建检查、Playwright 或拟真人工 QA。

## 浏览器与移动端验收

- Chrome Android 与 Safari iOS 是强制验收目标：移动端默认内置键盘、全量接入字符输入，不自动弹系统键盘。
- 其他浏览器允许降级：内置键盘优先，系统键盘可同时弹出。
- 系统键盘兜底能力保留在工程层，但首版 UI 屏蔽呼出按钮；不在答题区或键盘面板显示碍眼入口。
- 内置键盘不得挤压答题区：任何手机视口下，答题区高度占整体屏幕高度不得低于 60%。
- `virtualKeyboardPolicy` 只能作为增强能力，不作为跨浏览器核心方案。

## 题目质量验收

- `BL-009` 必须先抽样诊断，再根据诊断结论决定强制过滤规则。
- 诊断样本必须覆盖闯关中的整数乘法、整数除法、不同难度档、A03 相关 lane。
- 强制过滤规则上线前必须有统计证据和至少一组反例保护，避免误杀真正需要竖式训练的题。

## 竖式除法验收

- 长除法核心板必须覆盖整数除法、小数 ÷ 整数、小数 ÷ 小数、取近似、循环小数。
- 小数 ÷ 整数的小数点由系统预置，不作为学生输入格。
- 小数 ÷ 小数的扩倍转换是计算前训练格。
- 取近似只要求计算后填写保留 X 位结果，不额外填写第 X+1 位数字。
- 循环小数使用“非循环部分 + 循环节”结构化输入，并有长度上限。
- 所有需填写的过程格和训练格必须正确才通过。

## 错因与数据验收

- 统一失败面板必须说明“为什么最终答案正确但仍未通过”。
- 过程格错误按类别显示，不暴露具体错误值 / 正确值。
- 训练格错误显示具体类型、用户值和正确值。
- 错题本需要显示错误原因；历史记录 UI 不显示错误原因。
- 底层可保留 `failureReason`，用于未来分析。

## PM / QA 收口规则

- 每个 Phase 收尾前必须更新对应 phase 或 subplan 状态。
- 涉及 Backlog / Issue 生命周期变化时，先更新权威源，再回写 `Overview.md`。
- v0.5 收口时必须处理 `BL-009`、`BL-011`、`BL-010` 和 `ISSUE-067` 的归档或延期；`BL-012` 不参与 v0.5 收口，后续由 v0.6 承接。
- 涉及版本启动、收口、跨源写入时运行 `npx tsx scripts/pm-sync-check.ts`。
- 正式实现完成后按风险触发 `qa-leader`；release gate 前必须有 QA 结论。
