## 建议时序

```text
Phase 1：先完成开工对齐与共享架构门
  ↓
Phase 2：先做 BL-009 口算样例过滤诊断与规则
  ↓
Phase 3：再做 BL-011 内置键盘基建 + ISSUE-067 错因反馈
  ↓
Phase 4：最后做 BL-010 竖式除法 UI
  ↓
Phase 5：统一回归、规格回写、版本收口
```

Phase 2（`BL-009` 口算样例过滤）排在前面。原因是竖式除法 UI 上线前，应先确保竖式题样本与“必须通过竖式训练完成”的学习目标匹配。

Phase 3（`BL-011` 内置键盘 + `ISSUE-067` 错因反馈）排在 Phase 4 前面。原因是除法长除法 UI 的主要交互负担来自多格输入和严格错因反馈，先做共享输入 / 反馈能力能降低后续重复实现风险。

## 决策门

| 决策 | 影响文件 | 未决时处理 |
|---|---|---|
| 内置键盘是否能稳定覆盖 Chrome / Safari | Phase 1 开工对齐、Phase 3 子计划 | 不能只依赖 `virtualKeyboardPolicy`；Phase 1 定边界，Phase 3 细化自绘输入格 + 内置键盘状态写入和系统键盘独立兜底入口。 |
| 口算题过滤规则 | Phase 2 子计划、可能影响生成器 / campaign | 先抽样诊断，再决定强制规则；不在诊断前直接固化阈值。 |
| 长除法 UI 数据结构 | Phase 1 开工对齐、Phase 4 子计划 | Phase 1 只确认边界和预留接口；完整轮次、商位、乘积、余数、落位 / 补 0、训练格模型在 Phase 4 子计划中细化，未确认前不得进入实现。 |
| 循环小数输入与屏幕适配 | Phase 4 子计划 | 使用结构化“非循环部分 + 循环节”；循环节和答案长度设上限，超限题进入取近似题池。 |
| 错因数据记录范围 | Phase 3 / Phase 4 子计划、错题本 | 历史记录 UI 不展示；错题本展示；底层保留 `failureReason`。 |
| `BL-012` 版本归属 | Backlog / v0.6 版本包 | 不进入 v0.5；v0.6 启动时再展开后台查询权限、只读保护和生产入口。 |
