## 目标

在已完成代码审查、单元测试、浏览器回归的基础上，补做一轮**拟真人工 QA**：

1. 不只验证“功能是否存在”，还验证“真实用户是否能顺畅理解并完成操作”。
2. 对此前自动化未充分覆盖的深度体验用例，建立明确的**用户预期**、**判定标准**、**实际观察**和**QA结论**。
3. 重点确认解锁链、退出弹窗、竖式板交互这三类高风险体验点。

---

## 测试方法

### 一、拟真人工 QA 的工作定义

本轮测试不把“人工”理解为必须由人肉手点，而是采用以下方法替代传统人工 QA：

- 由 agent 扮演真实用户，按页面可见信息和可见反馈完成流程。
- 对必须隔离前置状态的场景，允许使用**测试存档 / fixture**准备环境，但**结论只依据真实 UI 行为**给出。
- 每条用例均记录：
  - 用户在该场景下的心理预期
  - 实际操作路径
  - 实际观察到的反馈
  - 与预期的差异
  - QA 判定（PASS / FAIL / RISK）

### 二、判定原则

- `PASS`：功能结果正确，且交互/反馈符合真实用户直觉。
- `FAIL`：结果错误，或关键反馈与规格/用户预期不一致。
- `RISK`：功能可用，但存在理解成本、教学价值不足、节奏不自然、视觉提示弱等体验风险。

### 三、证据形式

- Playwright 实际操作日志
- 关键页面截图
- localStorage / session 数据对比
- 代码实现对照（仅用于解释现象，不替代 UI 结论）

## 执行策略

### 1. 解锁链

- 使用新注册用户真实通关 A01 Stage 1。
- 通过实际答题推进，不直接写完成状态。
- 在每次通关后返回地图，检查：
  - 本路线下一关是否解锁
  - 并行路线是否未受干扰
  - Stage 2 是否仍保持锁定 / 正确解锁

### 2. 退出弹窗

- 在真实闯关中途触发 `✕`。
- 分别验证：
  - 文案是否足够阻止误退
  - “继续练习”是否保持现场
  - “退出”是否真的不写记录、不污染错题本、不增加 session

### 3. 竖式板

- 使用 A03 对应路线，优先通过真实入口进入。
- 对必须验证更高阶段的场景，允许使用 fixture 准备到对应解锁状态。
- 对每个场景检查：
  - 布局是否符合纸笔计算直觉
  - 焦点跳转是否流畅
  - 退位/进位提示是否可理解
  - 错误反馈是否保留足够的教学过程
