# 项目管理文档体系轻量化优化方案（以杜绝 token 浪费为目标）

> 创建：2026-04-19  
> 父计划：无（项目管理治理专项）  
> 设计规格：本文件即治理方案  
> 状态：✅ 完成（Phase 1 角色瘦身、Phase 2 归档分流、Phase 3 规则收缩均已落地）

---

## 一、前提：小型个人项目管理必须保留什么

目标不是“少写文档”，而是**在不牺牲可靠性的前提下压缩活跃上下文**。对当前这类小型个人项目，最低要求只有以下 6 类：

| 必要信息 | 为什么必须保留 |
|---|---|
| 项目总体背景概述 | 让 agent 知道项目是什么、长期问题是什么，避免只见局部任务不见方向 |
| 当前阶段目标 | 让 agent 知道“这轮为什么做”，避免在 backlog、历史计划、未来路线之间漂移 |
| 当前主线与当前状态 | 让 agent 知道“现在在干什么、进行到哪里、是否阻塞” |
| 当前活跃执行计划 | 让 agent 有唯一的执行入口，而不是读完整个历史计划树 |
| 当前生效的硬约束入口 | 让 agent 能找到真正约束当前工作的 spec / rule / constant，而不是靠记忆 |
| 当前开放问题 / 风险 | 让 agent 不重复踩坑；已关闭事项可归档，但开放项必须可见 |

除此之外的内容也有价值，但默认角色应从“活跃上下文”降级为“归档 / 追溯材料”。

---

## 二、现状问题

当前 token 浪费的根因不是文档数量本身，而是**活跃状态被多处重复承载**：

1. `Overview.md`、主计划、`Plan/README.md`、当前 Umbrella 子计划都在重复写“当前主线 / 当前状态 / 下一步 / 本阶段不做什么”。
2. 已关闭 issue、已完成阶段、同日重排全过程仍停留在活跃视图里，占用默认阅读成本。
3. `pm-sync-check` 的触发范围大于它的实际覆盖面，容易从“收口工具”滑成“默认仪式”。
4. 索引文档和事实源文档职责混淆，导致 agent 很难判断“这里是入口”还是“这里是权威”。

所以本方案要解决的不是“把文档删掉”，而是把文档改造成：

**主管分流、专人持有、关闭归档、活跃前置。**

## 三、目标结构

### 3.1 角色分工

| 角色 | 文档 | 责任 |
|---|---|---|
| 主管 / 控制面 | `ProjectManager/Overview.md` | 只提供项目背景、当前阶段目标、当前主线、当前状态、下一步、入口链接 |
| 执行事实源 | 当前活跃主计划 / 子计划 | 只提供范围、里程碑、阻塞、执行证据、当前决策结果 |
| 规格导航 | `ProjectManager/Specs/_index.md` | 告诉 agent 当前工作该查哪些生效 spec |
| 约束事实源 | 各 `Specs/*.md` | 持有具体硬约束，不由 Overview/Plan/README 复写 |
| 问题事实源 | `ProjectManager/ISSUE_LIST.md` | 持有开放问题状态；关闭事项默认进入归档区 |
| 索引 / 归档入口 | `ProjectManager/Plan/README.md` | 只登记计划文件、模板入口、归档入口，不承载当前活跃状态 |
| 历史追溯 | `ProjectManager/Reports/` | 承接回溯、复盘、机制设计历史，不作为默认 planning 入口 |

### 3.2 主管只同步活跃视图

`Overview.md` 只允许出现以下内容：

- 项目总体背景概述
- 当前阶段目标
- 当前主线
- 当前状态
- 下一步
- 入口链接

`Overview.md` 不再复写：

- issue 细节
- spec 细节
- 同日重排全过程
- 已关闭阶段的大段摘要
- 历史决策的完整推理

一句话：**Overview 负责指路，不负责替专人把内容再讲一遍。**

## 四、更新规则

只保留 4 条硬规则，避免规则本身再次膨胀。

### 4.1 先改权威源，再改主管

信息变化时，先改对应的专人文档；只有当这次变化影响了“当前活跃视图”时，才改 `Overview.md`。

### 4.2 主管只写结果，不写过程

`Overview.md` 只同步结果性结论，如：

- 当前主线变了
- 当前状态变了
- 下一步变了
- 背景口径变了

过程性推理、争论、重排轨迹一律放回 Plan / Reports。

### 4.3 索引只在生命周期变化时更新

`Plan/README.md` 和 `Specs/_index.md` 只在以下事件发生时更新：

- 新建
- 归档
- 生效状态变化
- 文件改名
- 链接关系变化

日常里程碑推进不要求同步索引。

### 4.4 关闭事项立刻退出活跃视图

issue / plan / 路线一旦关闭或废弃：

- 从活跃视图移出
- 进入归档区
- `Overview.md` 只保留一句结果性结论

关闭事项保留追溯价值，但不再占用默认 planning 视野。

## 五、信息路由矩阵

| 想知道什么 | 先看哪里 | 再看哪里 |
|---|---|---|
| 项目是什么、现在在干什么 | `Overview.md` | 当前活跃主计划 / 子计划 |
| 当前这轮要做什么 | `Overview.md` | 当前活跃主计划 / 子计划 |
| 这件事受哪些规格约束 | `Overview.md` | `Specs/_index.md` → 对应生效 Spec |
| 具体规则或数值是什么 | 当前活跃计划中的链接 | 对应 `Specs/*.md` |
| 当前有哪些开放问题 | `Overview.md` 中入口 | `ISSUE_LIST.md` |
| 为什么会有这条规则 / 这次重排 | `Overview.md` 中入口 | `Reports/` 或历史 Plan |

默认阅读路径压缩为：

**Overview → 专人文档 → 必要时再进归档。**

## 六、`pm-sync-check` 的新定位

`pm-sync-check` 不删除，但必须从“默认仪式”降回“收口工具”。

### 6.1 保留什么

- `_index.md` 完整性
- Plan / Spec 引用存在性
- 机械可验证的版本 / 状态漂移
- 少量真正跨源的硬约束一致性

### 6.2 去掉什么

- 对只读诊断、只看代码、只写方案默认 pre-flight
- 对纯 `Overview.md` 文案精简默认强制触发
- 对并未修改权威源关系的 session 机械性双跑

### 6.3 新触发规则

只在以下 3 类场景运行：

1. **跨源写入后**：同一轮改了 2 个及以上权威源 / 索引源
2. **里程碑收尾前**：准备声明“这一块完成”时
3. **生命周期变化时**：新建 / 归档 / 废弃 Plan、Spec、Issue 入口关系

因此，`pm-sync-check` 的角色应是：

**少量关键节点抽查一致性**，而不是每次开工都先完成一套仪式。

## 七、落地顺序

### Phase 1：先做角色瘦身

1. 把 `Overview.md` 收敛为活跃控制面
2. 删除 `Plan/README.md` 中重复的当前阶段状态段
3. 让主计划 / 子计划删除大段已关闭事项摘要和重排全过程复写

### Phase 2：再做归档分流

1. `ISSUE_LIST.md` 明确 open / archived 两个视角
2. 已关闭 issue 默认退出活跃阅读区
3. 重排历史、机制推理沉到 `Reports/`

### Phase 3：最后收缩规则

1. 改写 `CLAUDE.md` / `Plan/README.md` / `.cursor/rules/pm-sync-check.mdc` 中对 `pm-sync-check` 的触发表述
2. 把“什么变化更新哪里”压成一张短表
3. 只保留轻量抽查，不再叠加新的流程层

### 2026-04-19 完成回写

- **Phase 1 已完成**：`Overview.md` 已收敛为活跃控制面；`Plan/README.md` 已退回索引/模板入口；主计划、Umbrella 与当前实施子子计划已完成活跃状态去镜像。
- **Phase 2 已完成**：`ISSUE_LIST.md` 已形成 open / archive 双视角，并补充“默认读到这里即可”的活跃入口；关闭事项保留原始台账但退出默认阅读面；Phase 3 Umbrella 的同日重排历史已下沉到 `Reports/`。
- **Phase 3 已完成**：`CLAUDE.md` 与 `.cursor/rules/pm-sync-check.mdc` 已收缩为“关键节点抽查”；活跃计划中的旧“四处同步”表述已统一改成“按角色最小同步”；`Plan/README.md` 已补“发生什么 → 更新哪里”短表作为索引入口。
- **最终校验**：`pm-sync-check` 已全绿；当前文档体系已满足“主管分流、专人持有、关闭归档、活跃前置”的目标结构。

## 八、验收标准

如果本方案生效，后续一个新 agent 在进入项目时，默认只需要读：

1. `ProjectManager/Overview.md`
2. 当前活跃主计划 / 子计划
3. `Specs/_index.md` 指向的当前相关 spec

并且应满足：

- 默认活跃上下文不再包含已关闭事项长列表
- 当前状态不再在 4 份以上文档里镜像
- `Overview.md` 能在短时间内回答“项目是什么、现在做什么、下一步是什么”
- `pm-sync-check` 只在少量关键节点运行
- 文档精简后，项目管理可靠性不下降：背景不断、目标不丢、约束可查、开放问题可见、历史可追溯

---

## 九、结论

这次优化的核心不是“减少文档”，而是：

**减少活跃重复、保留权威单点、把历史退出默认视野。**

对这个项目来说，最优解不是推倒重来，而是把现有体系重排成：

**`Overview` 当主管，专人文档持有事实，关闭事项归档，`pm-sync-check` 只守关键口。**
