## 一、背景

v0.4 发布后收到真实用户反馈：

- 任意格子填 `0` 后，自动跳转到下一个格子时，下一个格子也会被自动填上 `0`。
- 退位格的正确过程值是 `-1`，但用户不知道这一内部表示，反复尝试填 `1`。

这两个现象都落在 A03 `vertical-fill` 的 legacy single-line `VerticalCalcBoard` 输入层。多位乘法的 `MultiplicationVerticalBoard` 使用真实输入框，不在本 hotfix 范围内。

## 二、根因

`VerticalCalcBoard` 的隐藏 input 当前存在两条字符入口：

- `onKeyDown`：消费数字、`-`、退格、`Enter`、`Tab`。
- `onChange`：读取 `event.currentTarget.value.slice(-1)`，继续调用同一个 `handleInput()`。

部分输入环境中，同一个 `0` 会先通过 `keydown` 写入当前格并触发跳格；随后同一个字符又以 `input/change` 形式到达。此时组件已经重渲染，活动格变成下一格，于是同一个 `0` 被第二次写入下一格。

`0` 最容易稳定暴露，是因为答案格接受 `0-9`，过程格也常接受 `0`；其它数字即使重复到达，也可能被下一格的过程格校验拦截。

## 三、目标与范围

### 目标

- 从结构上消除同一字符被双入口消费的可能。
- 保留桌面键盘、移动端软键盘、触摸切格、`Enter` / `Tab`、退格清除等既有输入能力。
- 把退位格的用户心智从“必须知道内部值 `-1`”调整为“按 `1` 表示退 1”，内部仍归一为 `-1`。
- 不改变 `vertical-calc-policy` 的显示、跳格、提交、判定、错题原因和过程 warning 规则。

### 非目标

- 不重做竖式板视觉结构。
- 不改 A03 生成器、难度档位、关卡结构或题目分布。
- 不改 `MultiplicationVerticalBoard`。
- 不新增存档字段或迁移。

## 四、方案

采用单一字符输入入口：

| 入口 | 职责 |
|---|---|
| `onInput` | 唯一消费字符 / 文本变更：数字、`-`、移动端软键盘、粘贴、软键盘删除。事件处理后立即清空 hidden input 的 DOM value。 |
| `onKeyDown` | 只处理非文本控制键：`Backspace`、`Delete`、`Enter`、`Tab`。不再消费数字或 `-`。 |
| `commitCellInput(raw)` | 唯一写格子状态的函数：根据当前格、运算符和已有过程值决定写入答案 / 过程值，并调用策略层跳格。 |

退位语义：

- 减法过程格输入 `1`：保存为 `-1`，表示“退 1”。
- 减法过程格输入 `0`：保存为 `0`，表示“不退位”。
- 减法过程格输入 `-`：保留临时 `-`，不跳格；继续输入 `1` 后保存为 `-1`。
- 其它运算的过程格规则沿用当前策略：加法 `0/1`，乘法 `0-8`。

## 五、联动检查

| 维度 | 风险 | 控制 |
|---|---|---|
| 桌面数字键 | 从 `keydown` 消费改为 `input` 消费后，焦点可能不跳 | e2e 覆盖 `999+888` 低档 / 中档焦点回归。 |
| 移动端软键盘 | 软键盘不稳定触发 `keydown` | `onInput` 成为唯一字符入口，软键盘路径更直接。 |
| 同一字符重复到达 | 再次出现跨格填值 | `onKeyDown` 不消费字符；同一字符只剩一条写入路径。 |
| 退格清除 | 移动端退格可能不走硬键盘 `keydown` | `onInput` 识别 `inputType='deleteContentBackward'` 时清当前格。 |
| `Enter` / `Tab` | 跳过 0 过程格能力丢失 | 保留在 `onKeyDown` 的控制键分支。 |
| 粘贴多字符 | 绕过逐格训练 | 多字符直接拒绝并清空 hidden input。 |
| 非法字符 | 残留在 hidden input，影响下次输入 | 每次 `onInput` 后都清空 DOM value。 |
| 策略层 | 输入重构误改判定语义 | 不修改 `vertical-calc-policy.ts`；只改 UI 输入 ownership。 |
