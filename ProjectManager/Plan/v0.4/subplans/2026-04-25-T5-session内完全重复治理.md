## 一、问题定义

Phase 3 诊断发现，部分闯关关卡在同一 session 内高频出现完全重复题。例如同一句题干在一关 12-20 道题中多次出现，玩家会明显感到题库重复。T5 只治理完全重复，不治理结构重复。

本任务要在玩家主要练习路径中减少同一 session 内的完全重复，同时不破坏生成器纯随机职责，不做跨 session 永久去重。

## 二、已确认决策

- 只治理完全重复。
- 不治理结构重复。
- 使用题型感知 duplicate signature。
- 在 session 出题层做 bounded retry。
- retry 上限 5 次。
- retry 5 次后仍重复时，仅开发环境记录诊断信息。
- 覆盖玩家主要练习路径：
  - campaign
  - advance
  - rank-match
- signature 只在单个 session 内生效，session 结束后清空。
- 诊断脚本输出同源 baseline vs simulated retry 的精确对比，不预先写死 60% 下降目标。

## 三、重复签名规则

### 3.1 闭合题干

题干本身已经包含具体题目实例，选项只是答案候选。

签名：`normalized prompt`

示例：

- `25 × 37 × 4 可以先变成哪一个？`
- `判断正误："小数除以小数，商一定是小数"`
- `把 0.75 化成最简分数，分母是几？`

这些题只要 prompt 相同，就算重复，即使 options 不同也不接受。

### 3.2 开放题干

题干只是通用任务，真正题目实例由 options 承载。

签名：`normalized prompt + canonicalized options`

示例：

- `下面哪个式子能先凑整？`
- `以下哪些说法正确？`
- `下面哪个算式的积比 300 大？`

这些题 prompt 相同但 options 不同可以接受；options 只是顺序不同则仍视为同一题。

### 3.3 不确定模板

无法稳定自动判断时，由对应 generator/helper 显式声明 duplicate signature policy：

- `closed-prompt`
- `open-options`

## 五、实现架构决策门

| 维度 | 决策 |

## 六、技术实现

### 6.1 新增 question-dedupe helper

建议新增 `src/engine/question-dedupe.ts`。

核心 API：

```ts
export const DEDUPE_RETRY_LIMIT = 5;

export function getDuplicateSignature(question: Question): string;

export function generateUniqueQuestion(params: {
  generate: () => Question;
  seen: Set<string>;
  context: {
    sessionMode: 'campaign' | 'advance' | 'rank-match';
    topicId?: TopicId;
    difficulty?: number;
    subtypeTag?: string;
  };
  retryLimit?: number;
}): Question;
```

`generateUniqueQuestion` 行为：

1. 生成一题。
2. 计算 duplicate signature。
3. 若 signature 未出现，加入 `seen` 并返回。
4. 若已出现，最多重抽 5 次。
5. 仍重复则接受当前题，并在开发环境输出诊断。

开发环境诊断建议：

```ts
if (import.meta.env.DEV) {
  console.debug('[question-dedupe] retry exhausted', {
    sessionMode,
    topicId,
    difficulty,
    subtypeTag,
    signature,
    retryCount,
  });
}
```

### 6.2 signature 归一化

`normalized prompt`：

- trim
- 连续空白归一
- 保留数学符号与中文标点

`canonicalized options`：

- 从 `question.data.options` 读取。
- 对每项 trim / 空白归一。
- 排序后 join。
- 选项顺序不同仍视为同一 options 集合。

题型感知 policy：

第一版可采用启发式 + allowlist：

- 若 prompt 包含具体数字、表达式、判断陈述，默认 `closed-prompt`。
- 若 prompt 属于开放题干 allowlist，使用 `open-options`。
- allowlist 初始包含：
  - `下面哪个式子能先凑整？`
  - `以下哪些说法正确？`
  - `下面哪个算式的积比 300 大？`

若后续发现误判，改为 generator 显式声明 policy。

### 6.3 campaign / advance 接入

入口：`src/store/index.ts` 的 `nextQuestion()`。

当前 campaign / advance 每次调用：

```ts
question = generateQuestion(session.topicId, difficulty, subtypeFilter);
```

改为：

- 首选方案：在 store runtime state 中维护 `sessionDuplicateSignatures: Set<string>`。
- 该 set 只存在于内存，不写入 `PracticeSession`、repository 或 localStorage。
- `startCampaignSession` / `startAdvanceSession` 初始化空 set；session 结束、放弃或切换模式时清空。
- `nextQuestion` 对 campaign / advance 调用 `generateUniqueQuestion`。

注意：

- 已答题保存在 `session.questions`，刷新恢复时 set 会丢失。当前范围不要求跨刷新继续去重。
- 若希望刷新恢复仍去重，可从 `session.questions` 重建 set；这不需要持久化新字段，优先作为可选增强。

### 6.4 rank-match 接入

段位赛题序在 `src/engine/rank-match/question-picker.ts` 中预生成，再由 `startRankMatchGame` 写入 `rankQuestionQueue`。

因此 rank-match 的去重应在 `pickQuestionsForGame` / `generateBucket` 内完成，而不是 `nextQuestion`。

实现建议：

- 在 `pickQuestionsForGame` 级别创建 `seen = new Set<string>()`。
- `generateBucket` 增加可选 `seenSignatures` 参数。
- bucket 内每次生成题目时调用 `generateUniqueQuestion`。
- primary / nonPrimary / review 三个 bucket 共用同一个 set，确保整局不重复。

### 6.5 诊断脚本模拟

`scripts/diagnose-phase3-question-quality.mjs` 增加两组统计：

- baseline：原始生成。
- simulated retry：同样 session 路径下套用 `generateUniqueQuestion`，retry limit = 5。

输出：

- `baselineAvgDuplicateExtra`
- `simulatedRetryAvgDuplicateExtra`
- `absoluteDrop`
- `relativeDrop`
- `retryExhausted`

结构重复继续单独输出为 observation，不作为失败项。

## 八、验收指标

- campaign / advance / rank-match 均接入 session 内 bounded retry。
- retry 上限为 5。
- signature 只在单 session 内生效。
- 结构重复不作为失败项。
- 诊断脚本输出同源 baseline vs simulated retry：
  - `baselineAvgDuplicateExtra`
  - `simulatedRetryAvgDuplicateExtra`
  - `absoluteDrop`
  - `relativeDrop`
  - `retryExhausted`
- 若某热点下降不足，必须通过 `retryExhausted` 说明题池容量不足，并拆出扩池任务，不直接判定机制失败。
- rank-match 不得高于 baseline。

## 九、风险与回滚

- 题型感知签名启发式可能误判开放/闭合题干；发现后优先为 generator 增加显式 policy。
- 小题池下 retry 无法完全避免重复，必须依赖 `retryExhausted` 诊断解释。
- 若 store 内 set 与刷新恢复冲突，优先从已答 `session.questions` 重建 set，而不是写入持久化字段。
- 回滚方式：移除 session / rank picker 对 `generateUniqueQuestion` 的调用，保留诊断脚本对重复基线的输出。
