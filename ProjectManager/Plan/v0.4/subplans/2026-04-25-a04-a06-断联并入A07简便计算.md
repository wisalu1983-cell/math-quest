## 一、背景

用户重新评估 A04「运算律」和 A06「括号变换」后，确认它们不足以作为玩家主界面的独立大题型。二者本质是 A07「简便计算」的前置知识：先认识运算律和括号变形，再进入简便计算的识别、变形、诊断。

因此 v0.4 需要做一次题型信息架构调整：在玩家体验层完全移除 A04/A06 的题型身份，并把 A07 会用到的运算律 / 括号变换能力迁入 A07，使其成为 A07 自有的低档知识点 lane，而不是让 A07 特例引用 A04/A06。

## 二、目标与范围

### 目标

- 玩家在游戏任何位置都不再看到 A04「运算律」/ A06「括号变换」作为独立题型。
- A07「简便计算」低档关卡新增两个知识点 lane：「运算律」和「括号变换」。
- 原 A07 低档基础应用 lane 保留在低档，作为前置知识之后的第一层应用练习。
- 两个知识点 lane 覆盖原 A04/A06 的所有子题型，并在 lane 内保持由易到难的小梯度。
- 后续新生成的运算律 / 括号变换知识点题，在数据身份和玩家语义上都彻底归属 A07「简便计算」。
- A07 直接拥有会用到的运算律 / 括号变换生成能力；A04/A06 作为大题型从主链路断联，只保留旧数据读取所需的 legacy 兼容。

### 非目标

- 不重写整个生成器体系。
- 不把 A04/A06 历史存档清空或丢弃。
- 不把旧 A04/A06 闯关 / 进阶进度折算为新 A07 lane 进度。
- 不在本 Phase 解决所有题型的选项干扰项问题；只处理随 A04/A06 断联并入必须一起收口的相关 UX/选项问题。
- 不顺手做 Practice 状态重置重构。

## 三、方案决策

### 1. Phase 定位

本任务不适合合并到原 Phase 2「题目质量与生成器诊断」，也不适合塞进原 Phase 3「交互设计与教学引导」。

原因：

- 它改变玩家可见题型数、主界面入口、关卡地图和段位/进阶题型范围。
- 它会影响 `BL-006` 运算律 UX 和 `BL-007` 中 A04/A06 相关选项题的处理方式。
- 如果先做重复题诊断或选项题扩容，再改题型 IA，诊断样本和验收口径会被重排。

结论：设为 v0.4 新 Phase 2。原 Phase 2 顺延为 Phase 3，原 Phase 3 顺延为 Phase 4，Practice 工程质量顺延为 Phase 5/待确认。

### 2. 玩家可见题型

玩家可见题型从 8 个变为 6 个：

- 基础计算
- 数感估算
- 竖式笔算
- 小数计算
- 简便计算
- 方程移项

玩家体验口径：

- A04/A06 作为题型在游戏任何玩家可见位置都不存在：不出现在 Home、Campaign、Advance、Profile、WrongBook、History、SessionSummary、Rank 或任何筛选 / 统计入口。
- 「运算律」「括号变换」可以作为 A07 内部知识点 lane 名出现，但不能带 A04/A06 编号、独立题型卡片、独立星级、独立进阶、独立段位赛主轴或独立统计。
- 旧数据展示时统一归入「简便计算」；必要时只作为二级知识点标签显示，例如「简便计算 · 运算律」，不得恢复为同级题型。

### 3. A07 新关卡结构（已确认）

A07「简便计算」闯关地图确认重组为：

| 阶段 | Lane | 来源能力 | 子题型覆盖 |
|---|---|---|---|
| 低档 | 运算律 | 原 A04 能力迁入 A07 | `law-identify`、`law-structure-blank`、`law-reverse-blank`、`law-simple-judge`、`law-counter-example`、`law-easy-confuse`、`law-concept-reverse`、`law-compound-law`、`law-distributive-trap`、`law-error-diagnose` |
| 低档 | 括号变换 | 原 A06 能力迁入 A07 | `bracket-remove-plus`、`bracket-remove-minus`、`bracket-add`、`bracket-division-property`、`bracket-nested`、`bracket-four-items-sign`、`bracket-error-diagnose` |
| 低档 | 基础简便应用 | 原 A07 低档 | `bracket-normal`、`extract-factor`；`decimal-two-step` 的低难度版本可作为低档后段或中档桥接 |
| 中档 | 简便识别与进阶执行 | 原 A07 中档 | `bracket-hard`、`extract-factor` 中档拆分、`decimal-two-step` 方法选择、`simplify-subtract` |
| 高档 | 综合变形与诊断 | 原 A07 高档 | `bracket-demon`、`extract-factor` 隐藏公因数、`decimal-multi-step`、`decimal-chain` |

注意：

- 「每个 A04/A06 知识点 lane 包含所有原子题型」不等于第一关随机打出所有难度。实施时必须在 lane 内按 difficulty 和 subtype 权重拆小梯度。
- 原 A07 低档不能整体后移。`bracket-normal` 和 `extract-factor` 是学生把前置规则用于简便计算的第一层应用，应继续留在低档。

S1 每条 lane 压缩为 2 个关卡，最终方案如下：

| Lane | 关卡 | 训练目标 | 合并子题型 |
|---|---|---|---|
| 运算律 | L1 律的识别与基本结构 | 识别用了什么律、判断基本适用性、填写等式结构 | `law-identify` + `law-simple-judge` + `law-structure-blank` |
| 运算律 | L2 反用、边界与常见陷阱 | 反向使用运算律，判断边界，识别分配律常见错误 | `law-reverse-blank` + `law-counter-example` + `law-concept-reverse` + `law-easy-confuse` + `law-compound-law` + `law-distributive-trap` + `law-error-diagnose` |
| 括号变换 | L1 去括号基础 + 添括号 | 练括号可以去掉/加上，式子仍等价；以低风险模板为主 | `bracket-remove-plus` + `bracket-remove-minus` 轻量模板 + `bracket-add` 基础模板 |
| 括号变换 | L2 变号与诊断 | 专练减号前括号、连减/连除、多项变号、错误诊断 | `bracket-remove-minus` 核心变号 + `bracket-add` 减号侧 + `bracket-division-property` + `bracket-four-items-sign` + `bracket-error-diagnose` |
| 基础简便应用 | L1 能不能简便 + 连减凑整 | 判断哪些式子可以简便，做基础凑整拆分 | `bracket-normal` |
| 基础简便应用 | L2 分配律凑整 + 近整拆分 | 提公因数、分配律凑整、接近整十/整百的拆分 | `extract-factor` low + `decimal-two-step` lite |

`bracket-nested` 不进入 S1 主体；如需出现，只能作为 L2 末段极少量拓展，优先放到中档或高档诊断。

完整模拟样题见 [`../phases/2026-04-25-a07-新地图完整模拟题目清单.md`](../phases/2026-04-25-a07-新地图完整模拟题目清单.md)。

### 4. 断联与数据身份策略

本 Phase 不要求立刻从类型系统或旧存档 schema 中删除 `operation-laws` / `bracket-ops`，但它们不再作为 internal source 被新主链路调用。它们的身份收窄为 legacy data id：只服务旧存档、旧历史、旧错题的读取和展示兼容。

推荐实现：

- 建立玩家可见题型能力，例如 `PLAYER_TOPICS` / `isPlayerVisibleTopic()`；所有玩家入口只使用 6 个玩家可见题型。
- A07 生成器直接拥有所需知识点能力；运算律 / 括号变换生成逻辑按 A07 lane / level 结构重写为 A07-owned knowledge modules，不从旧 A04/A06 生成器薄包装或直接 import。`generateQuestion('multi-step', ...)` 不再调用旧 A04/A06 topic generator。
- 新生成题必须满足：`topicId: 'multi-step'`，`data.kind: 'multi-step'`；如需区分内部知识点，使用 `knowledgePoint: 'operation-laws' | 'bracket-transform' | 'simplification'` 等 A07 内部字段。
- `data.kind: 'operation-laws' / 'bracket-ops'` 只允许出现在旧数据兼容路径，不作为新题主链路身份。
- 历史记录和错题中已有的旧 topicId 不删除，展示时映射为「简便计算」或「简便计算 · 运算律 / 括号变换」；但不参与新段位赛复习抽题。

### 5. 进阶模式口径

A07 进阶模式保留原先 A07 的子题型分配，不把 A04/A06 的知识点 lane 追加进进阶池。

决策理由：

- A04/A06 是进入简便计算前的前置知识，主要在 A07 闯关低档完成教学与固化。
- 进阶模式是通关后的熟练训练，应继续聚焦原 A07 的简便计算应用能力：能否简便、凑整、因数拆分、方法选择、错误诊断、多步综合。
- 若把原 A04/A06 全量混入 A07 进阶，会稀释进阶训练目标，并迫使 `ADVANCE_MAX_SUBTYPES = 4` 的现有抽题架构重做能力家族配比；本 Phase 不需要承担这个复杂度。

实施口径：

- `buildAdvanceSlots('multi-step', ...)` 的 A07 子题型分配原则保持原样。
- Advance 选题入口只展示玩家可见的 6 个题型；不展示 A04/A06 进阶卡。
- 旧 A04/A06 `advanceProgress` 保留为 legacy 数据，不参与玩家可见星级、段位赛入场、题型筛选或统计。
- 新 A07 进阶题只归属「简便计算」，不生成玩家可见的 A04/A06 题型记录。

### 6. 段位赛条件口径

少了 A04/A06 两个玩家可见大题型后，段位赛相关规则也必须同步收敛为 6 个玩家可见题型。

实施口径：

- 段位赛入场条件、资格缺口提示、星级要求文案，不再要求 A04「运算律」或 A06「括号变换」的独立星级 / 进阶进度。
- 段位赛主考题型排程、复习题型范围、弱项来源、题型筛选，都不能再把 `operation-laws` / `bracket-ops` 作为玩家可见主轴。
- A07「简便计算」承担简便计算相关训练；段位赛如需覆盖运算律 / 括号变换能力，应通过 A07 内部知识点或 A07 题目能力体现，不恢复 A04/A06 独立题型身份。
- 旧错题 / 历史记录中如果存在 A04/A06 topicId，只保留历史展示兼容，不参与新段位赛复习抽题或弱项统计。
- 验收需要覆盖 `entry-gate`、`match-state`、`question-picker`、Rank UI 文案和缺口提示。

### 7. 存档与迁移

必须遵守项目级存档原则：版本升级不得 `clearAll()`。

实施口径：

- 旧 A04/A06 `campaignProgress`、`advanceProgress`、`wrongQuestions` 保留在存档中，作为历史和兼容数据。
- 玩家可见入口不再读取旧 A04/A06 作为独立进度。
- 旧 A04/A06 进度不折算到新 A07 lane：完整通关和 partial progress 都只保留为 legacy 数据，不写入 A07 闯关 / 进阶进度。
- 段位赛入场要求、缺口提示和出题范围改为 6 个玩家可见题型，不再要求 A04/A06 独立星级。
- 必须递增 `CURRENT_VERSION` 到 5，新增 `migrateV4ToV5` 并登记到 `MIGRATIONS`。
- `migrateV4ToV5` 必须检查持久化的未完成段位赛 / 可恢复赛局状态；如果赛局中残留 `operation-laws` 或 `bracket-ops` 题型，应直接作废该未完成赛局，而不是让玩家恢复一局包含已隐藏题型的比赛。
- 如果 `GameProgress` 迁移函数无法直接触达 match-state 存储，实现时必须补对应 repository / session cleanup 路径，并在迁移测试中覆盖。

## 八、实现架构决策

本节是 Phase 2 开发前置门。本文前六节定义产品与 PM 口径；真正进入实现前，以下 D1-D5 必须作为代码结构、类型约束、迁移路径和验收条件落地。

### D1. 生成器架构：A07-owned knowledge modules

采用 C' 方案：A07 拥有自己的知识点生成模块，旧 A04/A06 生成器不再作为新题主链路依赖。

实施口径：

- 在 `multi-step` 生成器边界内新增 A07-owned knowledge modules，例如 `multi-step/knowledge/laws.ts` 与 `multi-step/knowledge/bracket-transform.ts`。
- 这些模块不是旧 `operation-laws.ts` / `bracket-ops.ts` 的薄包装，也不直接 import 旧 topic generator，而是按 A07 的 lane / level 结构重写生成逻辑。
- `laws.ts` 必须适配 A07 S1 运算律 lane 的小梯度：L1 聚焦识别、判断、基本结构；L2 聚焦反用、边界、混淆、复合和错误诊断。不能沿用旧 A04 生成器 `difficulty <= 5 / >= 6` 的两档映射。
- `bracket-transform.ts` 必须适配 A07 S1 括号变换 lane：L1 处理去括号基础 + 添括号；L2 处理变号、连减/连除、多项变号和诊断。
- 旧 A04/A06 生成器只保留 legacy 兼容和历史数据解释职责；如果实现阶段仍需要从旧代码吸收模板，应复制语义后重写为 A07 模块，不保留运行时依赖。

### D2. 类型系统：区分 legacy topic 与 player-visible topic

`TopicId` 可继续包含 `operation-laws` / `bracket-ops` 以兼容旧存档，但玩家主链路必须使用更窄的类型。

建议类型：

```ts
type LegacyTopicId = 'operation-laws' | 'bracket-ops';
type PlayerVisibleTopicId = Exclude<TopicId, LegacyTopicId>;
```

实施口径：

- Home、Campaign、Advance、Rank、Profile、History、WrongBook、SessionSummary 等玩家入口使用 `PlayerVisibleTopicId` 或由 `PLAYER_TOPICS` 推导出的可见题型集合。
- `Question.topicId`、旧错题、旧历史、旧进度等持久化读取路径可以继续接受完整 `TopicId`。
- 新生成题与新玩家行为记录不得写入 `operation-laws` / `bracket-ops` 作为主 topic id。
- 代码中如必须处理 legacy topic，应显式经过 `isLegacyTopic()` / `isPlayerVisibleTopic()` 分流，避免未来误把隐藏题型加回玩家入口。

### D3. 存档版本：必须升级到 v5

本 Phase 改变玩家可见题型数、段位赛入场条件、复习范围和未完成赛局恢复语义，属于需要显式迁移的存档结构变化。

实施口径：

- `CURRENT_VERSION` 必须从 4 递增到 5。
- 新增 `migrateV4ToV5` 并登记到 `MIGRATIONS`。
- 迁移必须保留旧 A04/A06 `campaignProgress`、`advanceProgress`、`wrongQuestions`、历史记录等 legacy 数据，不清空、不折算到新 A07 lane。
- 迁移必须重新对齐段位赛规则：6 个玩家可见题型作为新规则事实源。
- 当前 `match-state.ts` 会持久化未完成段位赛状态；迁移必须检查未完成段位赛 / 可恢复赛局。如果题目、题型排程或复习范围中残留 `operation-laws` / `bracket-ops`，应直接作废该未完成赛局。
- 作废未完成赛局只影响恢复中的比赛，不删除玩家历史成绩、错题或全局进度。

### D4. Campaign 三 lane 支持：不阻塞，但必须验收

代码类型上 `CampaignStage.lanes` 是数组，UI 也按 lane 列表渲染，因此 S1 放 3 条 lane 不是预设阻塞项。

验收口径：

- CampaignMap 在桌面与手机竖屏下都能展示 A07 S1 三条 lane。
- 允许滚动，但不能出现卡片过窄、标题截断、入口不可点击、阶段高度异常挤压后续内容等问题。
- 必须用真实游戏环境走查，不用 `?preview` 替代最终验收。

### D5. 子题型 ID：全量 namespace 化

原 A04/A06 子题型迁入 A07 后，不能把旧 id 原样并入同一个 `subtypeFilter` 命名空间。

实施口径：

- 原 A04 运算律能力全部改为 `law-*`：如 `law-identify`、`law-structure-blank`、`law-reverse-blank`、`law-simple-judge`、`law-error-diagnose`。
- 原 A06 括号变换能力全部改为 `bracket-*`：如 `bracket-remove-plus`、`bracket-remove-minus`、`bracket-add`、`bracket-division-property`、`bracket-error-diagnose`。
- 不允许使用裸 `error-diagnose`。若 A07 自身还需要简便计算诊断题，应使用独立命名，例如 `simplify-error-diagnose`。
- `campaign.ts`、生成器 `pickSubtype()`、测试用例、样题模拟和文档表格必须使用 namespace 后的新 id，避免实现时靠 lane 上下文隐式消歧。
